Cloud Pub/Sub Study Notes

1. Core Concepts

Publisher-Subscriber Pattern

Pub/Sub decouples message producers (publishers) from consumers (subscribers) using topics as message channels.

Key Components

ComponentDescriptionKey Properties
TopicNamed channel for messagesGlobally unique, region-agnostic
SubscriptionReceives messages from a topicOne-to-many from topic, each gets all messages
MessageData payload with attributesUp to 10MB, base64 encoded data
PublisherSends messages to topicsCan be any application or GCP service
SubscriberReceives messages from subscriptionsPull or Push delivery

Message Flow

  1. Publisher sends message to topic
  2. Pub/Sub distributes message to all subscriptions
  3. Subscribers receive messages via pull or push
  4. Subscriber acknowledges (ACKs) message
  5. Pub/Sub removes acknowledged messages
At-least-once delivery: Messages delivered at least once but may be duplicated. Design idempotent message handlers.

Exactly-Once Delivery

  • Enabled per subscription - not at topic level
  • Higher cost - additional processing overhead
  • Regional only - not available for global topics
  • Deduplication - based on message ID and publish timestamp
Architecture Limitation: Exactly-once delivery is only available for regional topics and subscriptions, not global ones.

4. Exam Traps

Common Gotchas

Ordering Key Misconception: Ordering keys don't guarantee global message order - only within the same key partition.
Regional vs Global: Exactly-once delivery and message ordering require regional topics. Global topics don't support these features.
Push Subscription Authentication: Push endpoints must validate JWT tokens from Pub/Sub to ensure message authenticity.

Configuration Traps

TrapRealityImpact
Small ACK deadlineCauses message redelivery loopsDuplicate processing
No dead letter queueFailed messages retry indefinitelyResource waste
Wrong retention periodMessages expire before processingData loss
Missing flow controlSubscriber overwhelmedHigh memory usage

Design Pattern Mistakes

  • Not designing for idempotency - At-least-once delivery means duplicates
  • Ignoring message size limits - 10MB max, use Cloud Storage for large payloads
  • Over-partitioning with ordering keys - Too many unique keys reduces throughput
  • Synchronous publishing - Use async publishing for better performance
Exam Tip: Questions about "guaranteed order" always require message ordering to be enabled and regional topics.
