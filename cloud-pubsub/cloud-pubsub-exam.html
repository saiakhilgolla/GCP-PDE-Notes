Cloud Pub/Sub - GCP PDE Exam Study Notes

Core Concepts

What is Cloud Pub/Sub?

Asynchronous messaging service for decoupling services. Publishers send messages to topics, subscribers receive from subscriptions.

Key Components

  • Topic: Named resource to which messages are sent
  • Subscription: Named resource representing stream of messages from topic
  • Message: Data and attributes sent by publisher
  • Acknowledgment: Signal that subscriber processed message

Message Flow

Publisher → Topic → Subscription → Subscriber → Ack/Nack

Architecture
Component Description Scalability
Topic Global resource, auto-partitioned Unlimited publishers
Subscription Per-topic, multiple allowed Auto-scales subscribers
Message Storage Distributed, replicated Petabyte scale

Delivery Models

Model How It Works Use Case
Pull Subscriber requests messages Batch processing, controlled rate
Push Pub/Sub sends to HTTP endpoint Serverless, Cloud Functions

Exam Traps
TRAP: "Exactly-once delivery" - Pub/Sub provides "at-least-once" by default. Exactly-once requires enabling exactly-once delivery on subscription AND idempotent processing.
TRAP: Message ordering only works with single publisher per ordering key. Multiple publishers break ordering guarantees.
TRAP: Push subscriptions require public HTTP endpoints or proper authentication. Pull subscriptions work behind firewalls.
TRAP: Dead letter topics must be in same project and region as source subscription. Cross-project dead letter not supported.
TRAP: Schemas are optional but once applied, all messages must conform. Schema evolution requires careful version management.
TRAP: Subscription filters run server-side but count against quota even for filtered messages. Not a cost optimization.
TRAP: Message retention of 7 days is hard limit. Extended retention requires BigQuery export or Cloud Storage.
TRAP: Seek to timestamp only works if message retention covers that time period. Cannot seek beyond retention.
Quick Cheatsheet
Push vs Pull Decision:
• Push: Cloud Functions, App Engine, public endpoints
• Pull: GKE, Compute Engine, controlled processing

Message Ordering

  • Enable on subscription creation
  • Set ordering key in message attributes
  • Same ordering key = same order guarantee
  • Reduces throughput, increases latency

Exactly-Once Delivery

  • Enable on subscription (preview feature)
  • Higher cost, lower throughput
  • Still need idempotent processing
  • Not available in all regions

Dead Letter Queues

  • Set max delivery attempts (5-100)
  • Configure dead letter topic
  • Same project/region requirement
  • Prevents poison messages

Schemas

  • Avro, Protocol Buffers supported
  • JSON schema for validation
  • Versioning with backward/forward compatibility
  • Client libraries auto-encode/decode
Question Patterns

Scenario: Need reliable message processing

Answer Framework:
1. At-least-once by default
2. Enable exactly-once if truly needed
3. Design idempotent processors
4. Use dead letter queues for failures

Scenario: High throughput vs. ordered processing

Decision Tree:
• Need order? → Enable ordering with keys
• Need max throughput? → Disable ordering
• Mixed? → Separate topics for ordered/unordered

Scenario: Push vs. Pull subscription choice

Requirement Recommendation Reason
Cloud Functions trigger Push Native integration
Batch processing Pull Control message rate
Private network Pull No public endpoint needed
Auto-scaling app Push Event-driven scaling

Scenario: Message persistence requirements

Decision Framework:
• ≤ 7 days? → Standard retention
• > 7 days? → Export to BigQuery/GCS
• Audit trail? → Enable message storage policy
• Replay capability? → Use snapshots

Common Exam Scenarios

  • IoT sensors → Pull subscription, batch processing
  • Real-time notifications → Push to Cloud Functions
  • Order processing → Ordered delivery with dead letter
  • Log aggregation → Pull with filtering
  • Microservices decoupling → Multiple subscriptions per topic
