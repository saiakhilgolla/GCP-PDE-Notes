1. Core Concepts

What is Cloud SQL?

Cloud SQL is a fully managed relational database service on GCP. Handles patching, backups, replication, and failover automatically. No infrastructure management required. Scales vertically (machine type) and horizontally (read replicas). Available as MySQL 5.7/8.0, PostgreSQL 12-15, or SQL Server 2019/2022.

Managed vs Manual Operations

Automatic:

  • Backups (daily automatic backups retained 35 days by default)
  • OS and database patching
  • Failover in HA configuration
  • Replication to standby or read replicas
  • SSL/TLS encryption in transit

Manual:

  • Schema design and optimization
  • Connection pooling strategy
  • Query tuning
  • Index creation
  • Maintenance windows (you choose the window)
Key Advantages Over Self-Managed Databases
  • 99.95% SLA for HA instances (regional)
  • Zero-downtime patching in maintenance window
  • Automatic failover to standby in seconds
  • Point-in-time recovery up to 35 days
  • Integrated IAM and encryption at rest (CMEK)
  • No server provisioning or OS patching needed

High Availability (HA) Architecture

HA Deployment Model

Regional Deployment (Multi-Zone):

  • Primary: Accepts reads and writes. Synchronous replication to standby.
  • Standby Replica: In different zone, same region. Receives writes synchronously. Cannot serve queries in normal operation.
  • Failover: Automatic if primary becomes unavailable. Standby promotes to primary (5-10 seconds typical).
  • Zonal Outage: Handled. Datacentre failure triggers failover.
  • SLA: 99.95% uptime SLA.

Cost: Standby replica charged at 50% of primary instance cost.

Replication Topologies

Type Sync Mode Purpose Read/Write? Region
Standby (HA) Synchronous Automatic failover Read-only (except on failover) Same region, diff zone
Read Replica Asynchronous Scale read capacity, analytics offload Read-only Same or different region
Cross-Region Replica Asynchronous Disaster recovery, data locality Read-only Different region (any)
Read Replica Behavior & Lag
  • Asynchronous replication: Primary does not wait for replica to confirm write. Lower latency on primary but potential lag on replica.
  • Replication lag: Typically milliseconds to seconds. Monitor via Replication Offset Lag metric in Cloud Console.
  • Promotion: Read replica can be promoted to a standalone instance (one-way, irreversible). Useful for failover in multi-region disaster recovery.
  • Connection string: Replicas have separate public IP. Applications must connect to replica endpoint explicitly for reads.
  • Cascading replicas: A replica can have its own replicas (MySQL/PostgreSQL only). Creates hierarchical replication.
