1. Core Concepts

What is Cloud HSM?

Cloud HSM is a managed, single-tenant Hardware Security Module that provides cryptographic capabilities with FIPS 140-2 Level 3 compliance. Your encryption keys never leave the HSM or are never accessible in plaintext to Google or any other user.

Key Principles
  • Single-Tenant: Dedicated HSM instance — not shared with other customers
  • Hardware-Backed: Cryptographic operations happen on dedicated hardware, not in software
  • FIPS 140-2 Level 3: High security standard — identity-based access controls, tamper detection
  • Key Material Never Leaves HSM: Private keys stay inside; only ciphertext or signatures leave
  • Automatic Key Backup: HSM state is replicated and backed up; loss of the physical HSM doesn't mean loss of keys
FIPS 140-2 Level 3 — What Does It Mean?
Level Requirements Cloud HSM?
Level 1 Basic encryption. Software-based acceptable. —
Level 2 Role-based access, audit logs, firmware verification. —
Level 3 Identity-based access, tamper detection, physical attack resistance. ✓ Cloud HSM
Level 4 Active tamper response, extreme environmental controls. —

Cloud HSM vs. Cloud KMS (Software)

Aspect Cloud HSM Cloud KMS (Software)
Backend Hardware Security Module (single-tenant) Google-managed software (multi-tenant)
Compliance FIPS 140-2 Level 3 FIPS 140-2 Level 1
Key Accessibility Never leaves HSM in plaintext Google manages key material (encrypted at rest)
Tenancy Single-tenant (dedicated to you) Multi-tenant (shared infrastructure)
Cost High (hourly + per-operation fees) Low (pay-per-operation)
Use Case High-compliance requirements (finance, healthcare, government), CA-certified keys General encryption, most GCP workloads
Availability Regional (limited zones) Multi-region, global

Key Operations Supported by Cloud HSM

  • Encrypt/Decrypt: Data encryption using keys stored on HSM
  • Sign/Verify: Digital signatures with asymmetric keys (RSA, ECDSA, EdDSA)
  • Key Generation: Create cryptographic key pairs directly on the HSM
  • Key Import: Bring your own keys into the HSM
  • Key Wrapping: Encrypt other keys with HSM keys (key hierarchy)
  • PKIX Operations: Full PKI support for certificate operations

High-Level Architecture Components

Component Purpose Details
HSM Cluster Physical hardware running cryptographic operations Geographically distributed for redundancy, automatic failover

Network & Access Flow

Typical Request Flow (Encrypt Operation)
  1. Application calls Cloud HSM client library: encrypt(data, key_name)
  2. Client library authenticates using service account credentials + mTLS
  3. gRPC request sent over encrypted channel to HSM endpoint
  4. HSM verifies IAM permissions for the service account
  5. Cryptographic operation performed on HSM hardware
  6. Ciphertext returned to client (key never leaves HSM)
  7. Operation logged to Cloud Audit Logs

Key Backup & Replication

  • Automatic State Replication: HSM state is continuously replicated to multiple physical HSM units
  • Encrypted Backup: Key material backed up in Google Cloud's encrypted storage; encrypted with an external key
  • Disaster Recovery: If an HSM fails, state can be restored from backup or replica
  • No Manual Export Needed: Unlike on-premises HSMs, Google manages failover automatically
