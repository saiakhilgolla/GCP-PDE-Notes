1. Core Concepts

What is Eventarc?

Eventarc is a fully managed, serverless event routing and delivery platform on GCP. It enables you to:

  • Ingest events from multiple sources (Audit Logs, Pub/Sub, Cloud Storage, Dataflow, etc.)
  • Route events based on filters and patterns
  • Deliver events to targets (Cloud Run, Workflows, Pub/Sub, Cloud Functions)
  • Transform and enrich events in transit using Cloud Run services
  • Decouple producers from consumers — event sources need not know about targets
📋 How Eventarc Works: The Flow

Step 1: Event Source

  • An event is emitted (e.g., file uploaded to GCS, API call logged in Audit Logs, message published to Pub/Sub)

Step 2: Event Matching

  • Eventarc receives the event and checks all active triggers
  • A trigger is a rule that matches events based on source, type, and optional attribute filters

Step 3: Routing & Transformation

  • If an event matches a trigger, Eventarc retrieves the event details (headers, payload, attributes)
  • Optional: Cloud Run service can enrich/transform the event

Step 4: Delivery

  • Event is delivered to the target (Cloud Run, Pub/Sub, Cloud Workflows, Cloud Functions)
  • Target processes the event synchronously or asynchronously

Step 5: Acknowledgment & Retry

  • If target returns success (HTTP 200-299), event is marked delivered
  • If target fails or times out, Eventarc retries (configurable retry policy)
  • Events that fail after max retries are sent to a Dead Letter Queue (DLQ) if configured

Triggers: The Matching Rules

A trigger is the configuration that matches events and routes them to targets. Key properties:

  • Event Filters: Match by event source, event type, and attribute filters (e.g., bucket name, resource name)
  • Service Account: Used to authenticate to the target and call Eventarc APIs
  • Destination: Cloud Run service, Pub/Sub topic, Cloud Workflows, or Cloud Function
  • Retry Policy: Max retries, backoff strategy
  • Dead Letter Queue: (Optional) Where failed events go after exhausting retries
🔍 Event Matching Logic: Filters & Patterns

Source-Based Filtering

  • Specify the event source: google.cloud.storage, google.cloud.pubsub.topic, google.cloud.audit.log, etc.

Type-Based Filtering

  • Match specific event types: google.cloud.storage.object.finalized, google.cloud.audit.log.v1.written

Attribute Filtering (CEL Expressions)

  • Use Common Expression Language (CEL) to match attributes
  • Example: match(bucket, "my-data-bucket") — only trigger on uploads to a specific bucket
  • Example: match(resourceName, "projects/my-project/datasets/sensitive*") — trigger on API calls to sensitive datasets
  • Supports operators: ==, !=, match(), startsWith(), endsWith()

Multiple Filters (AND Logic)

  • All filters must match for an event to trigger the destination
  • No OR logic at the trigger level — use multiple triggers or post-processing in the target
