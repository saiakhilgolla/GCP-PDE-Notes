1. Core Concepts

What is Cloud EKM?

External Key Manager (EKM) is a GCP service that enables you to manage encryption keys outside of Google's infrastructure while still leveraging GCP's encryption services. It allows your organization to maintain sovereign control over encryption keys in your own data centers or approved third-party Key Management Services (KMS).

Key Principles & How It Works

Core Principles:

  • Key Sovereignty: Keys remain outside GCP in your external KMS
  • CMEK Enhanced: Extends Customer-Managed Encryption Keys (CMEK) capabilities
  • Compliance: Meets regulatory requirements (GDPR, data residency laws)
  • Access Control: GCP must request key access via signed API calls
  • Dual Custody: Both GCP and customer have control points

How It Works:

  1. Customer stores encryption keys in external KMS (e.g., Thales, YubiHSM, on-premises)
  2. GCP services need to encrypt/decrypt data; they request key access via EKM API
  3. Customer's KMS evaluates the request against policies (access justifications, time limits)
  4. If approved, key material flows through encrypted channel to GCP service
  5. Data is encrypted/decrypted; key is not persisted in GCP
  6. All access logged for audit and compliance

Encryption Flow:

User Data → GCP Service (e.g., Cloud Storage) ↓ [Needs to encrypt/decrypt] ↓ EKM API → External KMS (customer-controlled) ↓ [Access Justification Check] ↓ Key Material (via secure channel) → GCP Service ↓ Encrypt/Decrypt Operation ↓ Key is NOT stored in GCP; returned/cleared

EKM vs Cloud KMS

Aspect Cloud KMS Cloud EKM
Key Location Google-managed HSMs in GCP Customer's external KMS (on-prem or 3rd-party)
Key Control Google + Customer (shared) Customer (sovereign)
Key Residency Multi-region by default Where customer's KMS is deployed
Access Justifications Not required Required for key requests
Compliance Use Case Basic encryption needs Data residency, sovereign control
Cost Model Pay per KMS operation No GCP key storage cost (external system cost)
Key Unwrapping Google has access Customer controls unwrapping
Availability Highly available (multi-region) Depends on external KMS availability

2. Architecture

EKM Architecture Components

Component Role Ownership
External KMS Stores actual encryption keys; enforces access policies Customer
EKM API Gateway for GCP services to request key access Google
Access Justification Service Evaluates requests against customer policies Customer (configured)
GCP Data Services Consume encrypted key material for data operations Google
Audit Logging Records all key access requests and decisions Google (logs in customer project)
EKM Connection Profile Configuration binding GCP to external KMS Customer
