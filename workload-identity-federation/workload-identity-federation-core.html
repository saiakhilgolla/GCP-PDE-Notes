1. Core Concepts

What is Workload Identity Federation?

Workload Identity Federation (WIF) enables external workloads (AWS, Azure, GitHub Actions, on-premises, Kubernetes) to authenticate to Google Cloud without using service account keys. Uses OpenID Connect (OIDC) tokens from external identity providers to obtain short-lived GCP credentials.

Key Benefit: Eliminates long-lived service account keys, reducing security risk and simplifying credential rotation.
How Workload Identity Federation Works
  • Step 1: External workload (e.g., GitHub Actions) requests OIDC token from its identity provider
  • Step 2: Identity provider issues OIDC token with claims (subject, iss, aud, etc.)
  • Step 3: Workload calls GCP Security Token Service (STS) with OIDC token
  • Step 4: STS validates token and returns short-lived GCP access token
  • Step 5: Workload uses GCP access token to call GCP APIs

Attribute Mapping & Condition Logic

Attribute Mapping: Extracts claims from OIDC token and maps them to GCP attributes. Uses JSON path expressions to select token claims.

google.subject — Maps to the unique subject claim (required)

attribute.{custom} — Custom attributes mapped from token claims

assertion.{header|payload} — Direct token claims access

Condition Logic: CEL (Common Expression Language) expressions restrict token acceptance.

Example: assertion.repository == 'org/repo' && assertion.environment == 'prod'

Service Account Impersonation

After obtaining GCP token via STS, workload calls IAM Service Account Credentials API to impersonate a service account using the token. Service account has IAM role bindings that grant specific permissions.

Flow: OIDC Token → STS Access Token → Impersonate Service Account → Use Service Account Permissions

Key Requirement: External workload identity must have iam.serviceAccountUser or iam.workloadIdentityUser role on the service account.

2. Architecture & Components

Workload Identity Federation Architecture
Component Role Key Responsibility
External Identity Provider Issuer Issues OIDC/SAML tokens with claims; validates workload identity
Workload (AWS/Azure/GH) Requester Requests OIDC token from IdP, then exchanges for GCP token
GCP Workload Identity Provider Configuration Defines trusted IdP, attribute mapping, condition logic
GCP STS (Security Token Service) Token Exchanger Validates OIDC token, returns short-lived GCP access token
Service Account Permission Holder Holds IAM roles; workload impersonates to gain permissions
Workload Identity Pool Grouping Logical container for one or more identity providers
