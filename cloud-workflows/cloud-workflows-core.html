1. Core Concepts

Cloud Workflows is a GCP service for orchestrating Google Cloud services and HTTP endpoints using declarative YAML workflows. It's serverless, event-driven, and designed for multi-step automation.

What is Cloud Workflows?

Cloud Workflows is a fully managed service that lets you orchestrate and automate Google Cloud services and APIs using a YAML-based workflow definition language. Key characteristics:

  • Serverless: No infrastructure to manage; automatic scaling
  • YAML-based: Declarative syntax with steps, branches, loops, and error handling
  • Service Integration: Native connectors for Cloud Functions, Cloud Run, BigQuery, Firestore, Pub/Sub, etc.
  • HTTP Callbacks: Wait for external systems to call back with results
  • Managed Execution: GCP handles retries, timeouts, state management
  • Pricing: Per-step execution; first 1,000 steps/month free
Workflow Execution Model

Cloud Workflows executes steps sequentially or in parallel, maintaining state across steps.

  • Stateful: Each step's output is available to subsequent steps via variables
  • Deterministic: Same input produces same output; no random state
  • Fault Tolerant: Built-in retry logic with exponential backoff
  • Long-Running: Can run for up to 1 year (default 1 month timeout per execution)
  • Callback Support: Steps can pause and wait for external callbacks to resume
Core Language Concepts
Concept Description Example
Steps Named execution units; execute sequentially unless parallelized - step_name: # step logic
Variables Mutable state; assigned with assign step; scoped to workflow - assign: [x: 5, y: 10]
Expressions ${...} syntax; evaluated at runtime; supports math, string ops, conditionals ${x + y}, ${json.fieldname}
Call Steps Invoke external services (Cloud Run, Functions, Pub/Sub, HTTP endpoints) - call: googleapis.compute.v1.instances.list
Return Values Captured from API calls in result variable result: step_result.body

Key Point: Step vs Connector vs Native Binding

Native Bindings: Direct calls to Google APIs (e.g., googleapis.compute.v1.instances.list). No extra cost; uses service account credentials.

Connectors: Pre-built integrations for third-party services (e.g., Salesforce, ServiceNow). Require connector setup and authentication.

HTTP Steps: Generic HTTP calls to any endpoint with custom auth; most flexible.

2. Architecture

Cloud Workflows architecture consists of the workflow definition, execution runtime, and integration points with GCP services.

Workflow Components
Component Role Details
Workflow Definition YAML blueprint stored in Cloud Storage or inline Defines steps, triggers, IAM permissions, and service account
Execution Runtime GCP-managed service that parses and executes workflows Handles state, retries, timeouts, parallelism
Service Account Identity for workflow execution; assumed at runtime Must have IAM roles for called services (Compute, BigQuery, etc.)
Triggers Event sources that initiate workflow execution Cloud Pub/Sub, Cloud Tasks, Cloud Scheduler, HTTP, manual
Connectors Pre-built integration modules for external services Salesforce, Jira, ServiceNow, Slack, etc.
