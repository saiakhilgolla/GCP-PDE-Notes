1. Core Concepts

What is Cloud Monitoring?

Cloud Monitoring (formerly Stackdriver Monitoring) is GCP's fully managed observability platform. It collects metrics from GCP resources, AWS resources, and on-premises systems. Provides real-time dashboards, alerting, and uptime monitoring.

Metrics Fundamentals

Metric: A time-series measurement of a resource. Has a resource type, metric type, labels (key-value pairs), and timestamp.

Metric Types:

  • Cumulative — monotonically increasing (e.g., request count)
  • Gauge — instantaneous measurement (e.g., CPU %, memory)
  • Delta — counts events in fixed time window (less common)

Metric Resolution: 60 seconds for most resources. Custom metrics: 1 second minimum interval.

Metric Retention: 6 weeks for all metrics at 1-minute granularity. Beyond 6 weeks, Monitoring downsamples. Cannot retrieve raw data older than 6 weeks.

Alerting Policy Essentials

An alerting policy defines WHEN to trigger an alert based on metric values or log content.

Components:

  • Condition: threshold or threshold + duration (e.g., CPU > 80% for 5 minutes)
  • Notification Channel: email, SMS, PagerDuty, Slack, webhook, etc.
  • Documentation: markdown that displays when alert fires
  • Autoresolution: optional automatic clearing when condition clears

Alert State Transitions:

  • OK → FIRING when threshold breached for duration
  • FIRING → OK if autoresolution enabled and threshold clears
  • Maximum 100 incidents open per alert policy at once
Custom Metrics

Application-level metrics written directly to Cloud Monitoring API. Allows tracking business logic, latency, errors, feature usage.

Write Path: Applications call monitoring.googleapis.com/v3/projects/*/timeSeries with ServiceAccount permission monitoring.metricWriter.

Metric Descriptor: Defined once per custom metric type. Specifies labels, value type (INT64, DOUBLE, STRING), unit.

Billing: Custom metrics billed per unique time series (resource + labels combination). Unbound cardinality is expensive.

Best Practice: Limit custom metric label cardinality. Example: don't use user_id as label if millions of users. Use bucketing or aggregation first.

Log-Based Metrics

Metrics automatically extracted from log entries by pattern matching.

Types:

  • Count — increments for each matching log entry
  • Distribution — histogram from values extracted by regex or JSON

Creation: In Cloud Logging, select logs → Create Metric → define filter + extraction pattern.

Latency: ~1 minute delay from log creation to metric availability.

Use Case: Extract error rates, latencies, business events from unstructured logs without application code changes.

Service-Level Objectives (SLOs)

Targets for system reliability. Defines acceptable performance/availability window.

SLO = objective. SLI = indicator. Error budget = tolerance for failures.

Example: SLO = 99.9% availability. SLI = availability measurement. Error budget = 0.1% downtime allowed per month (~43 seconds).

Cloud Monitoring SLO Features:

  • Built on request/window-based indicators
  • Tracks good requests vs. total requests
  • Generates compliance report monthly
  • Can alert when error budget approaching exhaustion
Uptime Checks

Proactive monitoring from multiple global locations. Pings endpoint and records latency + response status.

Protocols: HTTP/HTTPS, TCP, UDP, gRPC.

Frequency: 1 minute minimum interval between checks.

Locations: Check from 3+ geographic regions by default. Can customize.

Billing: Per-check billed monthly. Regional redundancy increases cost.

Use Case: Synthetic monitoring for public endpoints, SSL certificate validation, latency SLO tracking.

Workspaces

Organizational container for metrics, alerts, dashboards. One workspace per project. Can monitor multiple GCP projects + AWS + on-premises from single workspace.

Workspace Project Types:

  • Host Project — holds Monitoring workspace, receives all metrics
  • Monitored Projects — linked projects whose metrics feed into host workspace

Setup: Host project must have monitoring.metricWriter IAM in each monitored project.

Access Control: IAM in host project controls who sees all metrics. Cannot grant separate permissions per monitored project.

Dashboards

Custom visualizations of metrics. Shareable, filterable, support multiple chart types.

Chart Types: Line, Stacked Area, Stacked Bar, Gauge, Scorecard, Pie, Table, Heatmap.

Features:

  • Templated variables for filtering (e.g., ${region}, ${resource})
  • MQL (Monitoring Query Language) or legacy Monitoring Query Builder
  • Dashboard code exported/imported as JSON

Performance: 1-minute refresh latency. Real-time isn't available; design for observability, not ops theater.
