🗄️ GCP Database Migration Service — PDE Study Notes

1. Core Concepts

Database Migration Service (DMS) is GCP's managed service for migrating RDBMS workloads. It handles schema conversion, data movement, and live change capture — enabling zero-downtime migrations from on-prem or other clouds to Cloud SQL or AlloyDB.
Migration Types: Homogeneous vs Heterogeneous
Type Definition Examples Complexity
Homogeneous Source and target are same RDBMS engine MySQL → Cloud SQL MySQL, PostgreSQL → Cloud SQL PostgreSQL Low — schema stays identical
Heterogeneous Source and target are different RDBMS engines Oracle → PostgreSQL, SQL Server → MySQL, Oracle → AlloyDB High — requires schema conversion

Key insight: Heterogeneous migrations use Striped Schema Conversion Tool (SSCT) to auto-convert source schema to target dialect. Manual fixes often needed.

CDC (Change Data Capture)

Change Data Capture captures data modifications (INSERT, UPDATE, DELETE) from the source database after the initial full load.

Source DB CDC Method Behavior
MySQL Binary logs (binlog) DMS reads binlog stream, applies changes to target in real-time
PostgreSQL Logical replication slots DMS subscribes to logical decode stream
Oracle Supplemental logging + redo logs DMS reads redo log stream via LogMiner
SQL Server Native CDC (if enabled) or transaction log DMS reads change tracking or log scan

Exam point: CDC is always enabled during continuous migrations. Full load alone is not zero-downtime; CDC is what makes it zero-downtime.

Migration Phases
  1. Schema — Source schema assessed & converted (for heterogeneous). Published to target.
  2. Full Load — All existing data copied from source to target. Can take hours/days for large databases.
  3. Change Sync (CDC) — DMS captures & applies ongoing changes from source. Near real-time.
  4. Validation — Row counts, checksums, constraint checks. Ensure source ≈ target.
  5. Switchover — Application points to target. Source becomes read-only (if desired).

Continuous migrations run Full Load + Change Sync. One-time migrations run Full Load only.

Connection Profiles

DMS uses Connection Profiles to store source & target database credentials and network config.

Field Purpose Example
Host / IP Database server address 10.0.0.5 (private) or 203.0.113.42 (public)
Port Database port MySQL: 3306, PostgreSQL: 5432, Oracle: 1521
Username / Password Authentication Stored in Secret Manager (encrypted)
Database Default DB to connect (optional) For MySQL: "myapp_db"
SSL Encryption in transit Required if source enforces SSL

Exam note: Connection profiles can be reused across multiple migration jobs. Test them before creating a job.

Migration Jobs

A Migration Job is a resource that defines a single migration, linking source & target connection profiles.

Job Type Full Load CDC Use Case
Full Only ✓ ✗ One-time data copy. Acceptable downtime.
Continuous ✓ ✓ (auto-starts after full load) Zero-downtime migration. Production workloads.

States: CREATING → READY → RUNNING → COMPLETED | STOPPED | FAILED

2. Architecture & Components

DMS is a managed service. The control plane runs on Google infrastructure; the data path uses replication workers (running on Google Compute Engine in the same region as the target database).
High-Level Architecture
Component Description Location
Source Database On-prem, AWS, Azure, or other GCP region. RDBMS (MySQL, PostgreSQL, Oracle, SQL Server) External to GCP or different region
DMS Control Plane Manages job lifecycle, schedules tasks, logs, monitoring Google infrastructure
Replication Workers GCE instances that read from source, apply to target. Managed by DMS. Same region as target (Cloud SQL or AlloyDB)
Target Database Cloud SQL MySQL, Cloud SQL PostgreSQL, or AlloyDB GCP region (same as workers)
Network Connectivity Options
Method Source Access Security Best For
Public IP Source has public IP; firewall allows DMS IPs Lower (requires source to be Internet-facing) AWS RDS, Azure SQL (with public IP)
VPC Peering On-prem network peered with GCP VPC via VPN or Interconnect High (private network) On-prem database + VPN/Interconnect link
Cloud SQL Auth Proxy N/A (for target only) High (uses mTLS) Connecting DMS to Cloud SQL via IAM auth
Private IP (RFC 1918) Source on private subnet. DMS workers in same VPC/peered VPC. High GCP resources in same/peered VPC
