Vertex AI Endpoints

Core Concepts

What are Vertex AI Endpoints?

Fully managed service for deploying ML models for online prediction. Acts as a RESTful API gateway that routes prediction requests to deployed model versions.

Key Components

  • Endpoint: HTTP/HTTPS URL that accepts prediction requests
  • Deployed Model: Model version serving predictions on the endpoint
  • Traffic Split: Percentage allocation across multiple model versions
  • Machine Type: Compute resources (CPU/GPU/memory) for serving

Model Deployment Types

Type Use Case Scaling Cost Model
Dedicated Resources Production workloads Manual/Auto scaling Per-node hourly
Shared Resources Development/testing Automatic Per-prediction

Traffic Splitting

Deploy multiple model versions to same endpoint with percentage-based traffic allocation:

  • A/B testing between models
  • Canary deployments (5% new model, 95% stable)
  • Blue-green deployments (instant 0% → 100% switch)
Architecture

Request Flow

Step Component Function
1 Client Application Sends HTTP POST with prediction payload
2 Endpoint Load Balancer Routes based on traffic split configuration
3 Model Server Processes prediction request
4 Response Handler Returns predictions in JSON format

Exam Traps

🚨 Common Gotchas

Traffic Split Confusion

  • Trap: Thinking traffic split is load balancing
  • Reality: It's A/B testing - percentages must sum to 100%
  • Example: Cannot have two models at 75% each

Autoscaling Delays

  • Trap: Expecting instant scaling
  • Reality: 2-5 minute cold start for new replicas
  • Solution: Set appropriate min replicas for baseline traffic

Private vs Public Endpoints

  • Trap: Assuming private endpoints are cheaper
  • Reality: Same pricing, private is for security/compliance
  • Use private when: VPC-only access required

Model Version Management

  • Trap: Deleting model version without updating traffic split
  • Reality: Must set traffic to 0% before deletion
  • Best practice: Always use staged deployments

Shared vs Dedicated Resources

Aspect Shared (Trap) Dedicated (Reality)
Cost Model Always cheaper Cheaper for high volume
Performance Same latency More predictable
Scaling Manual control No scaling control

Request Size Limits

Critical: 1.5MB limit includes JSON overhead, not just raw data. Base64 encoding increases size ~33%.
