Exam Traps

Service Account Confusion

Trap: Service accounts are BOTH identities (can be granted roles) AND resources (roles can be granted on them)

Key: serviceAccountUser role lets you impersonate, serviceAccountTokenCreator lets you generate access tokens

Workload vs Workforce Identity

Workload Identity: GKE pods → Service Accounts (no key files)

Workforce Identity: External users → Google identities (OIDC/SAML)

Trap: Don't confuse these - workload = applications, workforce = humans

Policy Inheritance

Trap: You CANNOT deny inherited permissions with allow policies

Solution: Use Deny Policies or move resource to different project/folder

Order: Deny policies evaluated first, then allow policies are additive

Conditions Common Mistakes

Trap: Conditions on basic roles (Owner/Editor/Viewer) are NOT supported

Trap: Failed condition = denied access (not ignored)

Format: Use CEL expressions: request.time.getHours() >= 9

Custom Role Gotchas

Trap: Cannot grant permissions you don't have yourself

Trap: Some permissions cannot be included in custom roles (marked as "not supported")

Testing: Use --test-permissions flag to check effective permissions

Service Account Keys

Trap: JSON key files are security risk - prefer Workload Identity

Trap: Keys don't auto-rotate - must be manually managed

Best Practice: Use Application Default Credentials when possible
