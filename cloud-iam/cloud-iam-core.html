1. Core Concepts

IAM Fundamentals

Identity + Role = Policy
IAM controls who (identity) has what access (role) to which resources (resource).

Role Types

  • Predefined Roles: Google-managed, service-specific granular access
  • Custom Roles: User-defined, project or org level only
2. Architecture

IAM Policy Hierarchy

LevelScopeInheritance
OrganizationAll resources in org↓ Inherited by folders/projects
FolderAll resources in folder↓ Inherited by sub-folders/projects
ProjectAll resources in project↓ Inherited by all project resources

Policy Inheritance: Child resources inherit parent policies. More permissive wins (union of permissions).

Service Account Architecture

Service Account as Identity

  • Acts as principal
  • Granted roles on resources
  • Used by applications

Service Account as Resource

  • Can be impersonated
  • Has IAM policies
  • Controls who can act as SA

4. Exam Traps

Common Gotchas

  • Basic roles inheritance: Project Editor gets Editor on ALL resources, including new ones
  • Service Account impersonation: Need roles/iam.serviceAccountTokenCreator not just serviceAccountUser
  • Workload Identity: Requires BOTH GKE SA annotation AND IAM binding
  • Custom roles: Can only be created at project or org level, NOT folder
  • Deny policies: Always evaluated BEFORE allow policies, can't be overridden

Frequently Confused Concepts

Can attach SA to resourcesCan generate access tokens

Trick Question Patterns

  • "Minimum permissions" → Custom role, not predefined
  • "External developers" → Workforce Identity Federation, not Google accounts
  • "GKE pods access GCS" → Workload Identity, not service account keys
  • "Prevent access even for owners" → Deny policies
  • "Cross-project SA impersonation" → Grant roles at SA level, not project
