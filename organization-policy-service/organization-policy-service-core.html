GCP Organization Policy Service

Core Concepts & Architecture | PDE Exam Study Notes

1. Core Concepts

Organization Policy Service is a centralized policy engine that enforces compliance and governance across your GCP resource hierarchy. It works at the organization, folder, and project levels to define what resources can and cannot be created, configured, or used.

What is Organization Policy?

  • Definition: A centralized governance tool that enforces constraints on GCP resources across the entire organization
  • Scope: Applied at organization, folder, or project level
  • Enforcement: Prevents resource creation/modification that violates the policy
  • Deny-based: Works on a deny-by-default model (what is NOT allowed)
  • Separate from IAM: Orthogonal to access control; IAM grants permissions, Policy enforces what can be configured
Constraints vs Policies

A constraint is a reusable rule template (e.g., "restrict VM public IPs"). A policy is the enforcement of that constraint at a specific resource level (organization/folder/project). One constraint can be applied to multiple policies across the hierarchy.

Boolean vs List Constraints

Constraint Type How It Works Use Case Example
Boolean Simple ON/OFF toggle. Either the constraint is enforced or not. Binary compliance rules compute.disableSerialPortAccess (true = disabled, false = enabled)
List Whitelist or blacklist of allowed/denied values. Can specify multiple exceptions. Granular control with exceptions compute.restrictedImageFamilies (allow only certain image families)
Boolean Constraint Example 
Constraint: compute.disableSerialPortAccess
Enforced: true
→ Serial port access is DISABLED on all Compute Engine VMs (unless parent overrides)
List Constraint Example 
Constraint: compute.restrictedImageFamilies
Allowed List: ["debian-10", "debian-11", "ubuntu-2004"]
→ Only VMs launched with these image families; any other family is denied

Policy Inheritance & Resource Hierarchy

Organization Policy follows the GCP resource hierarchy: Organization → Folder → Project → Resource
Level Policy Scope Inheritance
Organization Applies to all folders and projects below Most restrictive level; policies cascade to children
Folder Applies to all projects and subfolders below Can override parent if relaxing; cannot loosen a parent's deny
Project Applies to resources within the project Most specific level; cannot override parent org/folder policy if stricter
Inheritance Rule: "Most Restrictive Wins"

If Organization denies resource X, a Folder or Project cannot allow X. Inheritance flows DOWN the hierarchy. Child policies cannot override parent constraints. The most restrictive policy in the chain is enforced.

2. Architecture

Organization Policy Service Components

Component Purpose Interaction
Constraint Predefined or custom rule template Defines what can be restricted
Policy Instance of a constraint applied at a resource level Enforces the constraint at Org/Folder/Project
Resource Hierarchy Organization → Folder → Project → Resource Determines policy scope and inheritance
Enforcement Point API call validation Checks constraint at resource creation/modification time
Custom Constraints User-defined constraints for org-specific rules Extended governance beyond built-in constraints

Data Flow: Constraint Enforcement

User Action (e.g., "Create VM with public IP") → API Request → Policy Engine (evaluates constraints at resource level and all parent levels) → Decision (Allow or Deny) → Response (Resource created or error)
Enforcement Timing

Constraints are evaluated at API call time. If a resource violates a policy, the API call is rejected immediately. The policy engine checks the specific resource level plus all parent levels in the hierarchy.

3. Key Configurations & Limits

Configuration Default/Limit Notes
Built-in Constraints 100+ predefined constraints Google-managed, updated regularly. Cover compute, storage, networking, IAM, etc.
Custom Constraints per Org 500 limit Organization-level custom constraints only. Requires Policy Admin role.
Policy Enforcement Delay ~15 minutes New/updated policies propagate across the system within this window
Conditional IAM (in Policy) Not applicable Organization Policy does NOT support conditional logic. Use Conditional IAM for that.
Dry Run Mode Available for testing Test policies without enforcement; see what would be blocked
Policy Audit Logging Built-in to Cloud Audit Logs All policy deny/allow decisions logged automatically

Common Built-in Constraints (Exam Scope)

Compute Engine Constraints
  • compute.disableSerialPortAccess — Disable serial port access on VMs
  • compute.requireShieldedVm — All VMs must be Shielded VMs
  • compute.requireOsLogin — Require OS Login for VM access
  • compute.restrictedImageFamilies — Whitelist allowed VM image families
  • compute.skipDefault
