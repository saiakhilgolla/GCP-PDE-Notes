GCP Dataproc - PDE Exam Notes

Core Concepts

Dataproc Overview

Dataproc is a managed Hadoop and Spark service on GCP. Provisions clusters in 90 seconds, integrates with other GCP services, and supports autoscaling.

Key Components

  • Master nodes: Run HDFS NameNode, YARN ResourceManager, Spark master
  • Worker nodes: Run HDFS DataNodes, YARN NodeManagers, Spark executors
  • Preemptible workers: Secondary workers using preemptible VMs (80% cost savings)

Execution Models

  • Persistent clusters: Long-running for interactive workloads
  • Ephemeral clusters: Job-specific, deleted after completion
  • Serverless Spark: No cluster management, fully managed

Exam Traps
Trap: Thinking preemptible workers can be 100% of cluster
Reality: Preemptible workers cannot exceed the number of primary workers. Need at least equal primary workers.
Trap: Using persistent disks for HDFS with preemptible VMs
Reality: Preemptible VMs lose local SSD data when preempted. Use Cloud Storage or persistent disks for important data.
Trap: Expecting Dataproc to handle all failure scenarios
Reality: Applications must handle preemption gracefully. Use checkpointing and fault-tolerant patterns.
Trap: Confusing Dataproc autoscaling with Spark dynamic allocation
Reality: Dataproc autoscaling adds/removes VMs. Spark dynamic allocation adds/removes executors within existing VMs.
Trap: Thinking workflow templates are just for scheduling
Reality: Templates define entire cluster + job lifecycle including creation, job execution, and deletion.
Trap: Using single master for production workloads
Reality: Single point of failure. Use HA mode (3 masters) for production persistent clusters.
Trap: Storing critical data only in local cluster storage
Reality: Cluster deletion loses all local data. Use Cloud Storage for persistent data storage.
Trap: Ignoring network costs between regions
Reality: Cross-region data transfer incurs egress charges. Keep compute and data in same region.
Quick Cheatsheet
Cluster Types:
• Ephemeral: Job-specific, auto-delete after completion
• Persistent: Long-running, manual lifecycle management
• Single-node: Master=Worker, for development only
Preemptible VMs:
• 80% cost savings vs regular VMs
• Max 24-hour runtime before forced preemption
• Cannot exceed primary worker count
• Use for fault-tolerant batch jobs only
Autoscaling:
• Primary workers only (not preemptible)
• Based on YARN pending memory
• Scale-up: 2-minute evaluation, 2-minute cooldown
• Scale-down: 5-minute evaluation, 10-minute cooldown
Serverless Spark:
• No cluster to manage
• Auto-scaling to zero
• Pay per batch usage
• Supports Spark SQL, PySpark, Scala
Integration Points:
• Cloud Storage: Default file system (gs://)
• BigQuery: Native connector for Spark
• Cloud SQL: JDBC connections
• Pub/Sub: Streaming source/sink
Workflow Templates:
• Define cluster + jobs as code
• Managed cluster lifecycle
• Parameter substitution
• Integration with Cloud Scheduler
Custom Images:
• Faster cluster startup
• Pre-installed dependencies
• Security patches
• Base on Dataproc images only
