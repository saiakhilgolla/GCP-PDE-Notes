1. Core Concepts

What is Cloud Memorystore?

A fully managed in-memory data store service. Provides Redis and Valkey instances for sub-millisecond caching, sessions, real-time analytics, and message queues.

Redis vs Valkey
Aspect Redis Valkey
Origin Original open-source in-memory store Linux Foundation fork of Redis (2024)
License BSD 3-Clause (dual license as of 7.0) Open source (SSPL/Elastic)
GCP Support Available in Cloud Memorystore Available in Cloud Memorystore (newer option)
API Compatibility Standard Redis commands Fully compatible with Redis commands
Performance Well-established, proven Comparable, community-driven improvements
Exam Context More common; default choice Know it exists; Valkey = Redis-compatible fork

Key Use Cases

  • Caching Layer: Cache database query results, API responses, expensive computations
  • Session Storage: Store user sessions for web apps; faster than database
  • Real-time Leaderboards: Sorted sets for rankings updated at scale
  • Rate Limiting: Track request counts per user with TTL
  • Pub/Sub Messaging: Lightweight event distribution between services
  • Task Queues: Resque, Celery, Bull use Redis for job management
  • Real-time Analytics: HyperLogLog for cardinality, bit operations for flags

Persistence Modes

RDB (Snapshotting)
  • Point-in-time snapshots of entire dataset
  • Compact binary format; fast to load
  • Scheduled periodically (e.g., every 6 hours)
  • Minimal CPU overhead during operation
  • Some data loss possible between snapshots
  • Good for disaster recovery backups
AOF (Append-Only File)
  • Logs every write command to disk
  • Durability closer to real-time; minimal data loss
  • Higher disk I/O than RDB
  • Log file can grow large; periodical rewriting needed
  • Slower recovery (replays commands on restart)
  • Better for compliance and audit trails

Eviction Policies

When memory is full, Memorystore evicts keys based on policy:

Policy Behavior Best For
noeviction Return error; refuse new writes Prevent data loss; tight SLA on memory
allkeys-lru Evict least recently used key from all keys General cache; working set smaller than memory
volatile-lru LRU only among keys with TTL Mixed volatile + permanent data
allkeys-lfu Evict least frequently used key Hit-rate sensitive workloads
volatile-lfu LFU only among keys with TTL Hybrid scenario with TTL keys
allkeys-random Random eviction from all keys Rare; testing only
