Clustering Best Practices

  • Column Order: Most-queried columns first (up to 4 columns)
  • High Cardinality: user_id, device_id, order_id work best
  • Combined Strategy: Partition by date + cluster by high-cardinality columns
  • Automatic Maintenance: Re-clustering happens transparently

4. Exam Traps

Common Gotchas

  • Partition Pruning Failure: Using functions like DATE(timestamp_col) in WHERE clauses prevents partition pruning
  • External vs Native: External tables cannot use streaming inserts, DML, or native partitioning
  • Clustering Order: First cluster column provides most pruning - order by query frequency, not alphabetically
  • Slot Allocation: On-demand projects share 2000 slots - not per query or per user
  • BigLake vs External: BigLake adds security features but requires Connection resources

Security Misconceptions

  • Authorized Views: Don't grant table access - users only need view access
  • Row-level Security: Requires BigLake tables for external data
  • Column-level Security: Use policy tags, not separate tables
  • VPC-SC: Only available in Enterprise+ editions
5. Quick Cheatsheet

Performance Optimization

  • Partition by low-cardinality time columns
  • Cluster by high-cardinality filter columns
  • Use LIMIT with ORDER BY for top-N queries
  • Avoid SELECT * - specify columns
  • Use approximate functions: APPROX_COUNT_DISTINCT

Cost Control

  • Set require_partition_filter=true
  • Use query labels for cost tracking
  • Preview query cost before running
  • Consider materialized views for repeated queries
  • Use BI Engine for sub-second dashboards

BigQuery ML Quick Commands

  • CREATE MODEL - Train model
  • ML.EVALUATE - Model metrics
  • ML.PREDICT - Make predictions
  • ML.FEATURE_INFO - Feature analysis
  • ML.EXPLAIN_PREDICT - Explainable AI
