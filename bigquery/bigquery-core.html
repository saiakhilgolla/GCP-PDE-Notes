BigQuery Core Concepts & Architecture

GCP Professional Data Engineer Study Notes

1. Core Concepts

BigQuery is a serverless columnar analytics database powered by Dremel execution engine and Capacitor storage format on Google's Colossus filesystem.

Internal Architecture

Component Function Key Details
Dremel execution engine Distributes SQL across thousands of workers Compiles query into tree of execution stages, each processing data slices independently
Capacitor storage Proprietary columnar format on Colossus Columns stored separately and compressed. Query reading 3 of 100 columns only touches 3% of data
Compute/Storage separation Dremel workers allocated on demand No cluster to provision. Storage persists independently of compute
Slot Unit of BigQuery compute (virtual CPU) On-demand projects get up to 2000 concurrent slots by default
Shuffle service Handles data redistribution between stages Runs off worker VMs. Workers don't need extra memory for shuffle

Table Storage Types

Native BigQuery Storage
  • Data stored in Capacitor format on Colossus
  • Fastest queries with full optimization support
  • Supports partitioning, clustering, materialized views, streaming inserts
  • This is the default and recommended option
External Tables
  • Data stored outside BigQuery (Cloud Storage, Bigtable, Google Sheets, Spanner)
  • BigQuery reads at query time - no storage cost in BQ but lower performance
  • Cannot use partitioning/clustering natively
  • Cannot use streaming inserts or DML
BigLake Tables
  • External data in Cloud Storage, S3, or Azure Blob via Connection resource
  • Adds row-level security and column-level policy tags on external storage
  • Object tables for unstructured data (images, video)

2. Architecture Deep Dive

Partitioning

Performance Impact
Partitioning physically divides table data into segments. When a query filters on the partition column, BigQuery reads only relevant partitions, reducing bytes processed and cost.
Partition Type When to Use Key Features
Ingestion-time (_PARTITIONTIME) No meaningful timestamp column in data Auto-assigns rows to daily/hourly partitions based on load time
Column partitioning Have DATE/TIMESTAMP/DATETIME column Partition by DAY (default), HOUR, MONTH, or YEAR
Integer range partitioning ID-based sharding patterns Partition by INT64 column across user-defined ranges
Partition Pruning Failures
  • Wrapping partition column in function: WHERE DATE(created_at) = '2024-01-15'
  • Filter inside subquery BigQuery can't resolve
  • Fix: Use direct literal comparison: WHERE order_date = '2024-01-15'

Clustering

Block Pruning: Clustering sorts data within partitions by up to 4 columns. BigQuery stores min/max metadata per block to skip blocks that can't contain matching rows.
Aspect Details
Column ordering First cluster column prunes most aggressively. Order from most-queried to least-queried
High cardinality benefit user_id, device_id, order_id benefit most. Low cardinality offers less benefit
Maintenance Automatic re-clustering as data is inserted. No manual maintenance required
Power pattern Partition by date + cluster by high-cardinality ID - benefits from both layers
Rule of Thumb
  • High cardinality columns (customer_id, user_id): Clustering
  • Low cardinality time-based columns (date, month): Partitioning
  • Status and region columns: Clustering after high-cardinality column

3. Key Configs & Limits

Partitioning Limits

Configuration Value Impact
Maximum partitions per table 10,000 Table creation fails if exceeded
require_partition_filter Boolean option Prevents accidental full table scans - returns error if no partition filter
partition_expiration_days User-defined Expired partitions automatically deleted

Pricing Models Comparison

Model Cost Structure Best For Break-even Point
On-demand $6.25 per TB processed Exploratory, ad-hoc workloads Below 100TB/month
Capacity (Standard/Enterprise/Enterprise Plus) Pay for committed slots whether used or not Large predictable workloads Above 100TB/month
Autoscaling slots Scale up during peak, down during low usage Variable workloads with peaks Baseline + burst pattern
Flex slots Purchase by minute (60s minimum) Burst workloads, month-end reports Short-term capacity needs

Reservation Assignment Model

Admin Project: Holds slot reservations and commitments ↓ Reservations: Named pools of slots (e.g., "production-2000-slots") ↓ Assignments: Link project/folder/org to a reservation ↓ Query Execution: Assigned projects consume from reservation ↓ Fallback: Unassigned projects use on-demand pricing
Idle Slot Sharing
Unused slots in one reservation can be temporarily lent to another reservation in the same admin project automatically. No configuration needed.

4. Exam Traps & Gotchas

Most Common Traps
  • Partition pruning failure: Wrapping partition column in function prevents pruning
  • CSEK in BigQuery: Does NOT exist. BigQuery only supports GMEK and CMEK
  • Policy tags behavior: Return ACCESS DENIED error, not NULL or masked values
  • Time travel limit: 7 days maximum. Use table snapshots for longer recovery windows
  • Random lookups: BigQuery takes seconds. Use Bigtable for millisecond lookups
  • Frequent updates: BigQuery is append-optimized. DML UPDATE scans whole table

DML Quotas & Best Practices

Operation Quota Best Practice
DML statements per day per table ~1,500 Use single MERGE instead of multiple UPDATE statements
MERGE statement counting Counts as 1 DML regardless of rows affected Bulk updates via staging table + MERGE

BigQuery ML Latency Trap

Critical Exam Point
  • BigQuery ML.PREDICT latency: Seconds-level
  • For sub-100ms real-time predictions: Export model to Vertex AI Endpoint
  • Alternative: Pre-compute predictions and serve from Bigtable/Memorystore

Security Gotchas

IAM Role Combinations
Critical Pattern: Service accounts querying BigQuery need BOTH:
  • bigquery.jobUser to create jobs
  • bigquery.dataViewer to read data
  • Alternatively: bigquery.user covers both
Table-Level Access Control
No Table-Level IAM
BigQuery does NOT support IAM roles at individual table level. Access control is at dataset level via IAM or below through authorized views.
Solution Pattern: 1. Create view in restricted dataset selecting only allowed columns/rows 2. Authorize the view on source dataset 3. Grant restricted users bigquery.dataViewer on restricted dataset only

5. Quick Cheatsheet

BigQuery ML Functions

Function Purpose Key Details
ML.EVALUATE Model evaluation metrics AUC-ROC, precision, recall for classifiers; MAE, RMSE for regressors
ML.PREDICT Run inference Returns predictions + probabilities. Seconds-level latency
ML.EXPLAIN_PREDICT Feature attributions Shapley values showing which features drove each prediction
ML.TRANSFORM Apply preprocessing Uses transformations defined in CREATE MODEL TRANSFORM clause
ML.FORECAST Time series predictions For ARIMA_PLUS models with confidence intervals

Advanced SQL Features

Materialized Views
  • Smart rewriting: BigQuery can use materialized view to answer base table queries
  • Incremental refresh: Only processes new/changed data
  • Limitations: Base table cannot be view, external table, or partitioned by ingestion time
  • Query requirements: Aggregation with GROUP BY, no non-deterministic functions
Time Travel vs Snapshots vs Clones
Feature Duration Cost Use Case
Time Travel 7 days No additional cost Recovery from accidental DELETE/UPDATE
Table Snapshots User-defined expiration Delta pricing Recovery beyond 7 days (read-only)
Table Clones Permanent until deleted Delta pricing Development/testing (writable)

6. Question Patterns

Signal Recognition
  • "Petabyte analytics" / "SQL queries" / "data warehouse" → BigQuery
  • "Complex SQL transformations with dependencies" → Dataform (not BigQuery Scheduled Queries)
  • "Row-level security by user identity" → Row access policies
  • "Column contains SSN - users should see XXXX" → Masking rules (not policy tags alone)
  • "Sub-100ms ML predictions" → Export to Vertex AI Endpoint (not BQ ML direct)

Common Scenario Patterns

Performance Optimization Scenarios
  • Query scans too much data: Add partitioning on date column + clustering on high-cardinality columns
  • Repeated dashboard queries slow: Use BI Engine for sub-second cached results
  • Expensive slot consumption: Switch to capacity pricing with reservations
  • Burst workloads: Use flex slots for temporary extra capacity
Security & Access Control Scenarios
  • Share data across organizations: Analytics Hub with linked datasets
  • Column-level access control: Policy tags with Fine-Grained Reader role
  • Data masking requirements: Masking rules on top of policy tags
  • Cross-dataset table access: Authorized views
Integration Scenarios
  • Real-time CDC from operational DB: Datastream to BigQuery
  • Streaming data with exactly-once: Dataflow with Storage Write API committed streams
  • Cross-cloud data querying: BigQuery Omni for AWS S3/Azure Blob
  • Large file uploads: Stage to Cloud Storage then load to BigQuery
Decision Framework
  1. Identify data access pattern: Analytics scans vs operational lookups
  2. Check scale requirements: Data volume and query frequency
  3. Evaluate latency needs: Seconds (BigQuery) vs milliseconds (Bigtable)
  4. Consider consistency requirements: Eventually consistent vs strong consistency
  5. Factor in existing ecosystem: SQL skills, BI tools, compliance needs
