Core Concepts
Storage Model & Architecture

BigQuery is a serverless columnar analytics database

Component Description Key Points
Dremel Engine Distributed SQL execution Compiles queries into execution trees, parallel processing
Capacitor Storage Columnar format on Colossus Columns stored separately, compressed, query only needed columns
Slot Unit of BigQuery compute Equivalent to virtual CPU, on-demand gets 2000 concurrent slots
Shuffle Service Managed data redistribution Handles data movement between query stages
Exam Key: Compute and storage completely decoupled - workers allocated on demand, no cluster to provision

Type Storage Location Performance Features

Partitioning & Clustering
Partitioning Deep Dive

Partitioning physically divides table data into segments

Partition Type Column Type Use Case Granularity Options
Ingestion-time _PARTITIONTIME pseudo-column No meaningful timestamp column in data Daily, Hourly
Column partitioning DATE, TIMESTAMP, DATETIME Natural date column exists DAY (default), HOUR, MONTH, YEAR
Integer range INT64 ID-based sharding patterns User-defined ranges (e.g., 0-999, 1000-1999)
Partition Pruning Failure: Wrapping partition column in function prevents pruning. Use direct literal comparison: WHERE order_date = '2024-01-15' ✅ not WHERE DATE(created_at) = '2024-01-15' ❌

Critical Partition Limits

  • Maximum partitions per table: 10,000
  • require_partition_filter option: Prevents full table scans
  • partition_expiration_days: Auto-delete expired partitions
Clustering Strategy

Clustering sorts data within partitions by up to 4 columns

Aspect Details Best Practice
Column Ordering First column prunes most aggressively Order from most-queried to least-queried
High Cardinality user_id, device_id, order_id Best candidates for clustering
Low Cardinality status, region (few distinct values) Less benefit, use after high-cardinality
Maintenance Automatic reclustering Background process, transparent
Power Pattern: Partition by date (reduces partitions scanned) + Cluster by high-cardinality column (reduces blocks scanned)

Rule of Thumb

  • Partitioning: Low cardinality time-based columns (date, month)
  • Clustering: High cardinality columns (customer_id, user_id first, then status/region)

BI Engine

In-memory analysis service for BI tools

What it Accelerates What it Doesn't
  • SQL aggregations and filters
  • Tables fitting in BI Engine reservation
  • Looker Studio, Tableau queries
  • Standard SQL from BigQuery API
  • Scripting/procedural language features
  • Complex JOINs across very large tables

Key Benefits:

  • Sub-second latency for cached queries
  • No slot consumption for BI Engine queries
  • Complementary to slot reservations

Model Export to Vertex AI

Export Workflow for Sub-100ms Serving

  1. Train model in BigQuery ML using CREATE MODEL
  2. Export model to Cloud Storage as TensorFlow SavedModel
  3. Import SavedModel into Vertex AI Model Registry
  4. Deploy to Vertex AI Endpoint for low-latency serving
Latency Trap: BigQuery ML.PREDICT has seconds-level latency. For sub-100ms per-user real-time predictions, export to Vertex AI Endpoint.
Advanced SQL Features
Materialized Views

Pre-computed, stored query results with auto-refresh

Key Benefits

  • Smart rewriting: Optimizer can use MV to answer base table queries
  • Incremental refresh: Only processes new/changed data
  • Auto-refresh: Within 5 minutes or on schedule

Requirements & Limitations

  • Base table cannot be view, external table, or ingestion-time partitioned
  • Query must be aggregation with GROUP BY
  • No non-deterministic functions
