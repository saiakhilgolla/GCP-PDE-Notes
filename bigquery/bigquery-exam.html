1. Core Concepts

Storage & Execution Model
  • Dremel: Distributed SQL execution engine, compiles queries into execution trees
  • Capacitor: Proprietary columnar storage format on Colossus filesystem
  • Slots: Unit of compute (virtual CPU), allocated on-demand
  • Separation: Complete decoupling of compute (Dremel) and storage (Colossus)
  • Shuffle Service: Managed data redistribution between query stages

4. Exam Traps

Common Gotchas & Tricks

❌ Partition Pruning Failures

  • Functions on partition columns: WHERE DATE(created_at) = '2024-01-01' may not prune
  • Partition filters in subqueries that can't be resolved at plan time
  • Dynamic SQL where partition filter isn't literal

❌ External Table Limitations

  • Cannot use native partitioning or clustering
  • No streaming inserts or DML operations
  • Performance slower than native tables
  • No materialized views

❌ Clustering Misconceptions

  • Column order matters - first column prunes most aggressively
  • Low cardinality columns (status, region) provide less benefit
  • Cost estimates are not guarantees
  • High cardinality (user_id, order_id) benefits most

❌ Pricing Model Confusion

  • Capacity pricing: pay for slots whether used or not
  • On-demand: charged only for data processed, not stored
  • Reservations apply to specific projects/folders
  • Flex slots have 60-second minimum billing

❌ BQ ML Limitations

  • Models trained in one dataset cannot directly access data from another
  • Some ML model types require specific SQL syntax
  • Model export limitations for certain types

5. Quick Cheatsheet

Dense Key Facts

✅ Storage Optimization

  • Partition: DATE/TIMESTAMP columns, max 10K partitions
  • Cluster: High cardinality columns, max 4, order matters
  • Best combo: Partition by date + cluster by user_id
  • Pruning: Direct column filters work, functions may fail

✅ Pricing Quick Rules

  • Break-even: ~100TB/month for capacity vs on-demand
  • On-demand: $6.25/TB processed, 2K slots default
  • Capacity: Pay for reserved slots, autoscaling available
  • Flex: Burst workloads, 60-second minimum

✅ Security Features

  • Authorized views: Query specific columns/rows without table access
  • Row-level security: Filter rows based on user identity
  • Column-level security: Policy tags for data classification
  • BigLake: RLS on external data in Cloud Storage

✅ BQ ML Essentials

  • CREATE MODEL: Train ML models with SQL
  • ML.PREDICT: Make predictions on new data
  • ML.EVALUATE: Assess model performance
  • Export: Models to Vertex AI or Cloud Storage

✅ BQ Omni

  • Multi-cloud: Query data in AWS S3 and Azure Blob
  • Same SQL: Standard BigQuery SQL interface
  • Data residency: Compute runs in same region as data
  • Pricing: Per-query based on data processed

6. Question Patterns

Decision Frameworks

🔍 Storage Type Selection

  • Native BigQuery: Best performance, need partitioning/clustering
  • External tables: Data must stay in original location
  • BigLake: Need row-level security on external data

🔍 Optimization Strategy

  • Time-based queries: Partition by DATE/TIMESTAMP
  • High-cardinality filters: Add clustering
  • Reduce costs: Use require_partition_filter
  • Mixed patterns: Partition + cluster combination

🔍 Pricing Model Choice

  • Sporadic usage: On-demand pricing
  • Predictable heavy usage: Capacity reservations
  • Burst workloads: Flex slots
  • Cost control: Set maximum bytes billed

🔍 Security Implementation

  • Column hiding: Authorized views
  • User-based filtering
