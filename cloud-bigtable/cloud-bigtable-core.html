Cloud Bigtable - GCP PDE Study Notes

1. Core Concepts

Cloud Bigtable is a fully managed, petabyte-scale, wide-column NoSQL database. Powers Google Search, Analytics, Maps, and Gmail.

Data Model Primitives

Component Description Key Point
Row Key Single string that uniquely identifies a row ONLY index - no secondary indexes
Column Family Named group of related columns Defined at table creation, stored together on disk
Column Qualifier Named column within a column family Dynamic - not defined at schema creation
Cell Intersection of row key + column family + column qualifier Stores single byte string value with timestamp
Tablet Unit of data storage and distribution Contiguous row key ranges, served by one node
Key Insight: All rows stored in sorted lexicographic order by row key. Adjacent row keys are physically co-located.
2. Architecture

Storage Architecture

Component Function Scaling Behavior
Bigtable Node Serves tablets, handles read/write requests Add nodes = linear throughput increase
Tablet Stores contiguous row key ranges Auto-splits as data grows
Colossus Underlying distributed filesystem storage Nodes hold metadata only, data in Colossus

Read/Write Paths

Write Path

  1. Client sends write to Bigtable node
  2. Node writes to commit log (WAL)
  3. Node updates memtable in memory
  4. Memtable flushed to SSTable in Colossus

Read Path

  1. Client sends read to appropriate node
  2. Node checks memtable first
  3. If not found, reads from SSTables
  4. Merges results and returns to client

Compaction Process

  • Minor Compaction: Memtable → SSTable
  • Major Compaction: Multiple SSTables → Single SSTable
  • Purpose: Remove deleted/expired cells, improve read performance
3. Row Key Design Patterns
EXAM CRITICAL: Row key design is the most tested Bigtable concept. Understanding good vs bad patterns is essential.

Good Row Key Patterns

Pattern Use Case Example
Reverse Timestamp Time-series, latest-first queries device_id#(MAX_LONG-timestamp)
Composite Key Multi-dimensional queries user_id#event_type#timestamp
Hash Salt Prefix Distribute writes evenly hash(key)[:4]#original_key
Reversed Domain Domain-based data com.google.www

Bad Row Key Patterns (Cause Hotspots)

Anti-Pattern Problem Fix
Sequential IDs (1,2,3...) All writes go to same tablet Add hash prefix or use UUID
Timestamp prefix Concurrent writes cluster together Use reverse timestamp
Low-cardinality prefix Limited tablet distribution Add high-cardinality prefix
Good Example - Reverse Timestamp: MAX_LONG = 9999999999999999 timestamp_us = int(time.time() * 1e6) row_key = "sensor_001#" + str(MAX_LONG - timestamp_us) # Result: newest records sort first Good Example - Hash Salt: import hashlib natural_key = "user_12345#2024-01-15#pageview" salt = hashlib.md5(natural_key.encode()).hexdigest()[:4] row_key = salt + "#" + natural_key # Result: writes distributed across all tablets
4. Key Configs & Limits
Configuration Default/Limit Notes
Max row key size 4 KB Lexicographically sorted
Max cell value size 256 MB But 10 MB recommended
Column families per table ~1000 max Recommend 1-10 for performance
Column qualifiers per row Millions Dynamic, not pre-defined
Cell versions (default) 1 Configurable GC policy
Nodes per cluster 1-1000+ Linear scaling
Min cluster size (prod) 3 nodes 1 node for dev/testing only
SSD storage per node 8 TB max Better performance than HDD
HDD storage per node 64 TB max Lower cost, higher latency

Garbage Collection Policies

  • Max Versions: Keep N most recent versions
  • Max Age: Keep versions newer than N days
  • Default: Keep 1 version (latest)
5. Cluster Sizing & Performance

SSD vs HDD Storage

Storage Type Latency Throughput Use Case
SSD Single-digit ms 10K QPS per node Low-latency applications
HDD ~20ms 500 QPS per node Batch analytics, archival

Replication Options

  • Single-cluster: Default, all data in one zone
