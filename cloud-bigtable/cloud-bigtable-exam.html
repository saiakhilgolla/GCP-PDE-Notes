1. Core Concepts

Data Model & Storage Architecture
Row Key = Only Index: Bigtable has NO secondary indexes. All queries must use row key or row key ranges.
ComponentDescriptionKey Points
Row KeyUnique string identifierOnly index, determines storage order
Column FamilyNamed group of columnsDefined at table creation, stored together
Column QualifierDynamic column namesCreated at write time, millions possible
CellRow+Family+Qualifier intersectionStores byte string with timestamp
TabletContiguous row key rangeUnit of distribution, served by one node
How Bigtable Works
  • Sorted Storage: All rows stored in lexicographic order by row key
  • Tablet Distribution: Tables split into tablets, each served by one node
  • Separation of Compute/Storage: Nodes serve requests, data stored in Colossus
  • Auto-scaling: Tablets split automatically, load balancing across nodes
  • Versioning: Multiple timestamped versions per cell

2. Architecture

Read/Write Paths

Write Path

  1. Client determines tablet location (cache or metadata lookup)
  2. Write logged to commit log (WAL)
  3. Write added to memtable (in-memory)
  4. Memtable periodically flushed to SSTable in Colossus

Read Path

  1. Client contacts appropriate Bigtable node
  2. Node checks memtable first
  3. If not found, reads from SSTables in Colossus
  4. Multiple SSTables may need merging

4. Exam Traps

Row Key Design Traps
TRAP #1: Using timestamp as row key prefix causes ALL writes to go to same tablet (hotspot)
Fix: Use reverse timestamp: MAX_LONG - timestamp
TRAP #2: Sequential IDs (1,2,3...) create write hotspots
Fix: Add hash prefix or use UUID
TRAP #3: Domain names as-is cluster all .com together
Fix: Reverse domain: com.google.www
TRAP #4: Low cardinality prefixes (status: active/inactive)
Fix: Use high-cardinality prefix like user_id
Architecture & Scaling Traps
TRAP #5: "Adding nodes immediately improves performance"
Reality: Takes time for tablets to rebalance across new nodes
TRAP #6: "Bigtable supports complex queries like SQL JOINs"
Reality: Only supports single-table operations by row key
TRAP #7: "Column families should be created dynamically"
Reality: Must be defined at table creation time
Replication & Consistency Traps
TRAP #8: "Multi-cluster replication provides automatic failover"
Reality: Application must handle cluster switching
TRAP #9: "Reads can be eventually consistent"
Reality: Single cluster is strongly consistent

5. Quick Cheatsheet

Row Key Design Patterns
GOOD PATTERNS:
  • Reverse timestamp: device_id + "#" + (MAX_LONG - timestamp)
  • Hash prefix: md5(key)[:4] + "#" + key
  • Composite key: user_id + "#" + event_type + "#" + timestamp
  • Reversed domain: com.google.www
BAD PATTERNS:
  • Timestamp prefix: timestamp + "#" + device_id
  • Sequential IDs: 1, 2, 3, 4...
  • Low cardinality prefix: status + "#" + user_id
Performance Quick Facts
  • Scaling: Linear with nodes (10K QPS per node)
  • Latency: 1-5ms for single-row operations
  • Storage: SSD for low latency, HDD for throughput
  • Hotspots: Caused by poor row key design
  • Column families: Keep to 1-10, defined at creation
  • Compaction: Automatic, merges SSTables
Use Cases & Anti-Patterns
IDEAL FOR:
  • Time-series data (IoT sensors, metrics)
  • High-throughput analytics
  • Real-time personalization
  • Financial trading data
NOT GOOD FOR:
  • Small datasets (<1TB)
  • Complex queries requiring JOINs
  • OLTP with transactions
  • Highly relational data
