1. Core Concepts

Cloud Run is a fully managed serverless platform that lets you deploy containerized workloads (Docker images) without managing servers. Pay only for execution time.
What is Cloud Run?

Cloud Run executes containerized code on-demand:

  • Serverless: No VM management; automatic scaling from 0 to 1000s
  • Containerized: Push any Docker image; runs in isolated containers
  • Stateless: Each request is independent; no session affinity
  • Event-driven: Triggered by HTTP requests, Eventarc events, Pub/Sub messages, Cloud Scheduler
  • Pay-per-use: Billed per 100ms of execution time (with free tier)
Containerized Workloads on Cloud Run
Container Image
Must be a valid Docker image; supports any base OS (Ubuntu, Debian, Alpine)
Port Binding
Service must listen on port 8080 (or specified via PORT env var)
Startup
Container must start in under 4 minutes
Request Timeout
60 minutes max (default 5 minutes); configurable per service
Statelessness
No local disk persistence across requests; use Firestore/Cloud Storage
Eventarc Integration

Eventarc is Google Cloud's managed eventing service that routes events from GCP services directly to Cloud Run services.

Key Points:

  • Event Sources: GCS (OBJECT_FINALIZE, OBJECT_DELETE), BigQuery (job completion), Pub/Sub, Audit Logs, Cloud Scheduler
  • CloudEvents Format: All events standardized in CloudEvents schema (type, source, id, time, data)
  • No Pub/Sub Setup: Replaces manual Pub/Sub topic → subscription → code pattern
  • Automatic Routing: Events delivered to Cloud Run service URL
  • Replay Capability: Eventarc does NOT support message replay (use Pub/Sub if needed)

Eventarc Trigger Components:

  • Event source (e.g., GCS bucket path)
  • Event type (e.g., google.cloud.storage.object.v1.finalized)
  • Service account (for authentication)
  • Destination (Cloud Run service URL)
Push Subscriptions (Pub/Sub)

Cloud Run can be a target for Pub/Sub push subscriptions, receiving events via HTTP POST.

Configuration:

  • Subscription Type: Push (Cloud Run receives POST requests)
  • Push Endpoint: HTTPS URL of Cloud Run service (publicly accessible or behind identity verification)
  • Authentication: Cloud Run IAM invoker role required; OIDC token in Authorization header
  • Message Format: Wrapped in Pub/Sub envelope with message and subscription fields
  • Retry Behavior: Pub/Sub retries failed pushes with exponential backoff
Concurrency Model

Concurrency: Number of simultaneous requests a Cloud Run instance can handle.

Default Concurrency
80 requests per instance
Max Concurrency
1,000 requests per instance
When Scaling Occurs
New instances created when incoming requests exceed concurrency limit
Tuning Strategy
Lower concurrency for long-running tasks; higher for quick operations

Use Case: High-concurrency applications where many workers would otherwise maintain database connections; serverless environments where new connections created per invocation.

Min Instances & Cold Starts
Min Instances
Minimum number of instances always running (even with 0 traffic)
Default Min Instances
0 (scales to zero when idle)
Why Use Min Instances?
Eliminate cold start latency; maintain warm connections (DB, cache)
Cost Implication
Min instances incur continuous billing even idle; factor into cost model
Cold Start Latency
~100-500ms for typical containerized services (JVM slower)
VPC Connector

VPC Connector: Enables Cloud Run instances to access private resources in a VPC network (CloudSQL, Memorystore, private APIs).

Key Details:

  • Purpose: Route Cloud Run outbound traffic through a VPC network
  • Resources Accessible: CloudSQL, Memorystore, Compute Engine, Cloud KMS (private)
  • Setup: Create Serverless VPC Connector in project; link to Cloud Run service
  • IP Range: Connector uses CIDR block for routing (e.g., 10.8.0.0/28)
  • Performance: Small latency overhead (~20-50ms); worth it for private DB access
  • Billing: Metered by vCPU-hours; ~$0.10 per vCPU-hour

Example Use Case:

Cloud Run service needs to connect to a private CloudSQL instance (no public IP). Deploy VPC Connector in same region; attach to Cloud Run service. Service can now query CloudSQL via internal IP.
