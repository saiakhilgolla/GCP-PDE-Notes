GCP PDE: GKE & Cloud Composer

Core Concepts, Architecture & Exam Prep

1. Core Concepts

GKE is Google-managed Kubernetes for containerized workloads. Cloud Composer is Apache Airflow on GKE for orchestrating data pipelines. This section covers the fundamentals of each service.

GKE Fundamentals

What is GKE?

Google Kubernetes Engine (GKE) is a managed Kubernetes service that automates deployment, scaling, and operations of containerized applications. Key points:

  • Fully managed — Google handles control plane, upgrades, patches
  • Multi-zone / multi-region — High availability options built-in
  • Auto-scaling — Cluster autoscaler scales nodes; HPA scales pods
  • Workload Identity — Secure, key-less authentication to GCP services
  • Binary Authorization — Enforce only signed, verified container images run on cluster

Node Pools

What are Node Pools and when to use them?

A node pool is a subset of nodes within a GKE cluster with identical configuration (machine type, image, disk size, taints/labels). Single cluster can have multiple pools.

  • Use Case 1: Workload Isolation — Separate pools for compute-intensive vs memory-intensive tasks; use taints/tolerations to schedule pods
  • Use Case 2: Cost Optimization — Mix on-demand nodes with preemptible/spot nodes; use node selectors to route cheap workloads to preemptible pool
  • Use Case 3: Different Machine Types — GPU pool for ML training, CPU pool for serving, ARM pool for cost reduction
  • Use Case 4: Rolling Updates — Create new pool with updated image; drain old pool; delete old pool

Default pool: Created with cluster; nodes labeled with cloud.google.com/gke-nodepool: default-pool

GKE Modes: Autopilot vs Standard

Feature GKE Autopilot GKE Standard
Node Management Google manages nodes (creation, patching, maintenance) You manage nodes, machine types, OS images
Cost Model Pay per pod per hour + GKE service fee Pay per node + compute resources used
Auto-scaling Automatic node scaling based on workload Manual or cluster autoscaler (requires config)
Security Defaults Network Policy, Workload Identity, Binary Authorization enabled by default Optional, must enable
OS Updates Automatic, no user action User schedules updates via maintenance windows
Node Pools Google-managed, limited customization Full control; create custom pools
Best For Cloud Composer, stateless microservices, hands-off ops Cost control, GPU/TPU workloads, custom OS
Composer 2 uses GKE Autopilot by default. Composer 1 used Standard. Autopilot workers scale to zero when idle — major cost savings vs Composer 1.

Workload Identity

How does Workload Identity work?

Workload Identity allows Kubernetes Service Accounts (KSAs) to impersonate Google Service Accounts (GSAs), eliminating need for key files.

  • Setup: Bind KSA to GSA via iam.gke.io/gcp-service-account annotation
  • Mechanism: GKE injects environment variables + token file; workload uses to request short-lived credentials from Google's STS
  • Security: No keys in pod; credentials expire hourly; audit trail via Cloud Audit Logs
  • Exam Trap: Must enable Workload Identity at cluster creation or via update. Composer 2 comes enabled by default.
# Bind KSA to GSA: kubectl annotate serviceaccount my-ksa \ iam.gke.io/gcp-service-account=my-gsa@PROJECT_ID.iam.gserviceaccount.com

Namespaces

Namespace isolation in GKE

Namespaces are logical clusters within a physical cluster — used for multi-tenancy, environment separation, RBAC scoping.

  • Default namespaces: default, kube-system, kube-public, kube-node-lease
  • Create namespace: kubectl create namespace production
  • Resource quotas: Limit CPU, memory, pod count per namespace
  • Network Policy: Restrict ingress/egress per namespace
  • Cloud Composer: Each Composer environment creates a dedicated GKE namespace for its workload (scheduler, workers, web server)

Binary Authorization

What is Binary Authorization?

Binary Authorization is a GKE security feature that enforces container image verification — only signed images from approved registries can be deployed.

  • How it works: Admin creates KMS key; signs attestations (statements that image is approved); GKE enforcer checks attestation before pulling image
  • Key concepts:
    • Attestor: Entity that signs attestations (typically CI/CD pipeline)
    • Attestation: Signed claim that image has passed security gates
    • Policy: Rules defining which attestors are trusted, which registries are allowed
  • Exam Trap: Binary Authorization is cluster-wide — affects all pods. Cannot exempt a pod unless policy explicitly allows it.
  • Composer integration: Composer environments using custom images must ensure those images are attestable, or Binary Authorization enforcement fails at task execution time.
