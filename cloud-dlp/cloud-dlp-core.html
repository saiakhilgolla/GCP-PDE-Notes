1. Core Concepts

What is Cloud DLP?

Cloud Data Loss Prevention (DLP) is a Google Cloud service that discovers, classifies, and protects sensitive data across Google Cloud and on-premises resources. It uses machine learning to identify personally identifiable information (PII), credentials, and other sensitive data types.

Key Capabilities
  • Data Discovery: Scan storage systems to find sensitive data
  • Classification: Identify data using built-in and custom info types
  • De-identification: Remove or mask sensitive data
  • Risk Analysis: Quantify re-identification risk
  • Monitoring: Track DLP activities and create alerts

Info Types

Info types are patterns that Cloud DLP uses to identify sensitive data. They define what constitutes sensitive information in your organization.

Info Type Category Examples Use Case
Built-in PII EMAIL_ADDRESS, PHONE_NUMBER, PERSON_NAME Out-of-box detection for common PII
Credentials API_KEY, AWS_CREDENTIALS, PRIVATE_KEY Detect exposed secrets
Financial CREDIT_CARD_NUMBER, BANK_ACCOUNT Payment card and financial data
Custom Info Types Employee IDs, internal reference numbers Organization-specific patterns
Custom Info Type Creation
  • Regex Pattern: Define patterns using regular expressions
  • Dictionary Matching: Match against word lists
  • Hybrid: Combine patterns with context clues
  • Likelihood: Adjust confidence scoring (VERY_UNLIKELY to VERY_LIKELY)

Core Operations

Inspection

Scans content to find matches against configured info types. Returns findings with location, matched text, and likelihood score.

  • Batch inspection: Cloud Storage, BigQuery, Datastore
  • Streaming inspection: Real-time API calls
  • Sample-based: Check % of data to reduce costs
De-identification

Removes or obfuscates sensitive data from content. Used in inspection jobs to transform matched findings.

  • Redaction: Remove text entirely
  • Masking: Replace with symbols (e.g., X, *)
  • Pseudonymization: Replace with token or deterministic hash
  • Bucketing: Generalize values (e.g., age ranges)
Pseudonymization

Replaces sensitive data with reversible tokens, enabling analytics while protecting identity. Requires a cryptographic key or dictionary for reversal.

  • Deterministic: Same input always produces same token
  • Reversible: Can be decrypted with the correct key
  • Supports FFX (Format-Preserving Encryption): Token maintains original format
Risk Analysis

Measures re-identification risk for quasi-identifiers. Quantifies privacy exposure after de-identification.

  • K-anonymity: At least k records share the same quasi-identifier values
  • L-diversity: At least l distinct sensitive attribute values in each group
  • T-closeness: Distribution of sensitive values close to overall distribution
  • Delta-presence: Bounds presence of individuals in dataset

2. Architecture & Components

DLP Architecture Overview

Component Function Integration
DLP API Core service for inspect and de-identify operations REST/gRPC, called directly or via Cloud Functions
DLP Job Monitoring Tracks batch job progress and completion Cloud Logging, Cloud Pub/Sub notifications
Storage Connectors Interface with Cloud Storage, BigQuery, Datastore Automatic scanning, batching, parallelization
Templates Reusable inspection and de-identification configs Store patterns, transformations, settings as YAML
Stored Infotypes Library of custom info types for org Reference by name in inspection configs

Data Flow: Inspection + De-identification

  1. Source: Data in Cloud Storage, BigQuery, or via API
  2. DLP Inspection: Content scanned against info types
  3. Finding Generation: Matches reported with confidence scores
  4. Transformation: De-identification rules applied to findings
  5. Output: Transformed content written to destination
  6. Logging: Results logged to Cloud Logging

BigQuery Integration

DLP with BigQuery Scan
  • DLP can scan BigQuery tables directly without exporting
  • Automatically handles ARRAY and STRUCT columns
  • Results include table, column, row position
  • Supports sampling by row count or percentage
  • Can exclude columns by name pattern
BigQuery Export Formats for DLP
Format Best For Use Case
CSV Human readability, Excel import De-identified exports for non-technical users
JSON (newline-delimited) Human-readable inspection, schema preservation Inspect semi-structured data, nested objects
Avro Dataflow intermediate storage, schema evolution Streaming processing, interchange formats
Parquet Downstream Spark, BQ re-import, compression Analytics pipelines, efficient columnar storage
ARRAY and STRUCT Handling
  • ARRAY: Ordered list of same-type values. DLP inspects each element.
  • STRUCT: Named fields of mixed types. DLP traverses nested structure.
  • UNNEST: Flattens arrays for row-level analysis in DLP results
  • DLP reports findings with array index or struct field path
