🗄️ AlloyDB for PostgreSQL: PDE Study Guide

1. Core Concepts

What is AlloyDB?

AlloyDB is Google Cloud's fully managed, PostgreSQL-compatible relational database with analytical capabilities. It combines OLTP (online transactional processing) and OLAP (online analytical processing) in a single system.

Key Innovation: Proprietary columnar engine alongside PostgreSQL row engine. Queries automatically route to the optimal engine.
Columnar Engine Basics

The columnar engine stores data column-by-column rather than row-by-row. This enables:

  • Faster analytical scans (read only needed columns)
  • Better compression ratios (similar data types compress together)
  • Parallel processing of large aggregations
  • Automatic query routing without manual hints

When used: Queries with aggregate functions (SUM, AVG, COUNT), GROUP BY, large table scans. NOT used for point lookups or small result sets.

PostgreSQL Compatibility Layer

AlloyDB accepts standard PostgreSQL protocol and SQL syntax. You can:

  • Migrate existing PostgreSQL applications with minimal code changes
  • Use PostgreSQL drivers and tools (psql, pgAdmin, DBeaver)
  • Run PL/pgSQL stored procedures
  • Leverage PostgreSQL extensions (PostGIS, pgvector, UUID)

Exception: Not 100% compatible. Some advanced features unavailable (e.g., custom C extensions, certain replication modes).

High Availability (HA) Architecture

AlloyDB provides built-in HA across zones with zero data loss:

  • Synchronous replication: Primary and 2 standby replicas in different zones
  • Automatic failover: Detects primary failure, promotes standby in ~30 seconds
  • Read-write endpoint: Always points to current primary (updated during failover)
  • RPO = 0: Recovery Point Objective. No data loss on failover
  • RTO < 1 minute: Recovery Time Objective

Read Pools Deep Dive

Read pools allow horizontal scaling of read capacity across regions without increasing write latency.

Characteristics:
  • Separate replicas: Not the same as HA replicas. Can be in different region.
  • Eventual consistency: Lag typically 100ms-500ms depending on write volume.
  • Dedicated endpoint: Route reads to read pool via separate connection string.
  • Per-query routing: Smart driver detects read intent and routes automatically (Psycopg3).
  • Cost: Additional nodes billed separately. Cheaper than larger primary.

Use case: Heavy analytics on fresh-but-not-real-time data, reporting dashboards, large batch reads.

AlloyDB Omni (On-Premises & Edge)

AlloyDB Omni brings AlloyDB to your own data center or hybrid cloud environment.

  • Same PostgreSQL compatibility and columnar engine
  • Run on your own infrastructure (on-premises, VMware, Kubernetes)
  • Sync with AlloyDB clusters in Google Cloud (optional)
  • Fully managed by you (not SaaS). No Google management.
  • Use case: data residency requirements, low-latency local access, hybrid deployments
