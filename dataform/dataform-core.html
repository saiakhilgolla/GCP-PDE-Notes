1. Core Concepts

What is Dataform?

Dataform is a Git-integrated data transformation platform on Google Cloud that enables teams to manage data workflows using SQL and assertions. It provides version control, dependency management, and automated testing for data pipelines.

  • Git-native: All code stored in Git; version history and collaboration built-in
  • SQL-centric: Write transformations primarily in SQL with SQLX syntax
  • Dependency resolution: Automatically determines execution order based on table references
  • Data quality: Built-in assertions validate data integrity
  • Orchestration: Integrated scheduling and execution via Cloud Scheduler or Cloud Composer
SQLX: The Dataform SQL Extension

SQLX extends standard SQL with metadata and control flow instructions specific to Dataform. It compiles down to executable SQL statements.

Core SQLX Directives

Directive Purpose Example
config() Set table/view properties: type, partition, cluster, tags config({type: 'table', partition_by: 'DATE(created_at)'})
ref() Reference another Dataform object; creates explicit dependency SELECT * FROM ${ref('upstream_table')}
source() Reference external data source outside Dataform SELECT * FROM ${source('raw', 'events')}
declare Define variables for use in SQLX blocks declare cutoff_date = '2024-01-01'
pre_operations SQL executed before main transformation pre_operations { DELETE FROM ${ref('table')} WHERE date < ... }
post_operations SQL executed after main transformation completes post_operations { CALL `project.dataset.proc_cleanup`() }
-- Example SQLX file (mymodel.sqlx) config { type: 'table', materialized_as: 'view', tags: ['daily', 'critical'], } declare cutoff = TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY); SELECT id, name, created_at FROM ${ref('source_table')} WHERE created_at >= cutoff
Dependencies & Graph Execution

Dataform automatically tracks data lineage and executes models in dependency order. The execution graph ensures downstream tables are built only after their upstreams complete successfully.

  • Implicit dependencies: ref() calls inside SQL automatically register dependencies
  • Explicit dependencies: Use depends_on in config for non-SQL edges (e.g., external job completion)
  • Circular dependency detection: Dataform compilation fails if cycles are detected
  • Topological sort: Execution engine orders tasks to respect all dependencies
  • Parallel execution: Independent branches execute concurrently when possible
// model_a.sqlx config { type: 'table' } SELECT 1 as id // model_b.sqlx (depends on model_a) config { type: 'table' } SELECT id * 2 FROM ${ref('model_a')} // model_c.sqlx (depends on model_b) config { type: 'table' } SELECT id * 3 FROM ${ref('model_b')} // Execution order: model_a → model_b → model_c
Assertions: Data Quality Testing

Assertions are queries that validate data correctness. They run after model execution and fail the pipeline if violations are detected, preventing bad data from propagating downstream.

Assertion Types

Type Use Case Syntax
Row-level assertion Validate individual rows; returns rows that fail WHERE condition IS FALSE
Table-level assertion Check aggregate properties; returns count of violations SELECT COUNT(*) FROM ${ref('table')} WHERE ...
// assertion_no_nulls.sqlx (row-level) config { type: 'assertion' } SELECT * FROM ${ref('users')} WHERE id IS NULL OR email IS NULL // assertion_row_count.sqlx (table-level) config { type: 'assertion' } SELECT CASE WHEN row_count < 1000 THEN ERROR('Expected >= 1000 rows') END FROM ( SELECT COUNT(*) as row_count FROM ${ref('daily_events')} ) // On compile: assertion runs after 'daily_events' table creation // If query returns rows, entire run fails and is marked as failed
Incremental Tables

Incremental tables build new data only for updated rows or new partitions, rather than recomputing the entire table each run. This dramatically reduces compute and cost for large append-only fact tables.

Incremental Pattern

config { type: 'incremental', uniqueKeys: ['id'], partitionBy: 'DATE(created_at)', } SELECT id, name, created_at, revenue FROM ${source('warehouse', 'transactions')} %{ if (incremental()) { WHERE created_at >= _dbt_internal_dbt_max_partition } %}
  • Full-refresh mode: Clears table and rebuilds from scratch (first run or manual trigger)
  • Incremental mode: Appends only new/modified rows since last run
  • uniqueKeys: Column(s) that define row uniqueness; used for upsert logic
  • Merge strategy: Dataform uses MERGE to upsert or insert-only depending on config
  • incremental() function: Conditional SQL; true only on incremental runs
  • Max partition tracking: Dataform stores last processed partition timestamp
Tags & Metadata

Tags organize models and control which subsets of the graph are compiled and executed. Useful for separating critical, daily, and weekly pipelines.

  • Tag assignment: config({ tags: ['daily', 'critical', 'pii'] })
  • Tag inheritance: Models inherit tags from parent config in directories
  • Selective execution: Run only models matching tag filter via CLI or UI
  • Tag queries: --tags daily,critical includes both daily AND critical
  • Common patterns: frequency (daily/weekly), criticality (critical/non-prod), compliance (pii/public)
