1. Core Concepts

What is Pub/Sub Lite?

Pub/Sub Lite is Google Cloud's zonal, low-cost alternative to Pub/Sub optimized for high-volume, predictable streaming workloads. It provides:

  • Fixed, predictable pricing (no per-API-call charges)
  • Lower latency than standard Pub/Sub
  • High throughput with capacity planning
  • Zonal deployment (not multi-region by default)
  • Built-in partitioning and scaling
Zonal vs Regional Architecture

Zonal Deployment (Pub/Sub Lite)

  • Data stored within a single zone (e.g., us-central1-a)
  • Lower latency, lower cost
  • Single point of failure within that zone
  • Replication within zone only
  • No cross-region failover

Regional Deployment (Standard Pub/Sub)

  • Data replicated across multiple zones in a region
  • Higher availability and durability
  • Slightly higher latency and cost
  • Automatic failover within region
  • Suitable for critical, mission-critical applications
Partitions in Pub/Sub Lite

Partitions are the fundamental unit of throughput and parallelism in Pub/Sub Lite.

  • Each partition is independent — messages are distributed across partitions by publish key
  • Ordering guarantee: messages published with the same key to the same partition maintain order
  • Throughput scales linearly — each partition provides a fixed publish/subscribe rate
  • Publishers hash message keys to determine target partition
  • Subscribers consume from one or more partitions (subscriber is assigned partition(s))
  • Partition count is immutable after topic creation — plan carefully
Capacity & Throughput

Pub/Sub Lite capacity is measured in publish and subscribe throughput units, not individual messages.

  • Publish throughput: measured in MiB/sec
  • Subscribe throughput: measured in MiB/sec
  • Capacity is allocated per partition
  • Each partition has independent publish and subscribe quotas
  • Storage capacity is separate, measured in GiB
  • Exceeding capacity results in throttling (publish backoff, subscribe delay)

2. Architecture & Components

System Architecture Overview
Component Role Key Details
Topic Named feed of messages Contains N partitions; immutable after creation
Partition Unit of parallelism Ordered, independent message stream
Subscription Named consumer group Reads from all partitions of a topic; tracks offset per partition
Publisher Sends messages Routes by key to partition; batches for throughput
Subscriber Receives messages Pull or push delivery; tracks progress per partition
Admin API Management Create/delete topics, subscriptions, update capacity
Data Flow & Message Routing
Publisher Key = "user_123" ↓ Hash(key) mod partition_count → Partition 2 ↓ Message appended to Partition 2 ↓ Subscriber(s) receive from Partition 2 in order ↓ Subscriber acks offset

Key Points:

  • Hash function is deterministic — same key always routes to same partition
  • If key is null/empty, partition selected round-robin
  • Subscription offset tracking is per-partition
  • One subscriber instance typically handles 1 partition (parallelism = partition count)
Zonal Replication & Durability
  • Messages are replicated within the zone to multiple physical storage nodes
  • Default replication factor: 3 copies within the zone
  • Durability: 99.99% within-zone availability (not 99.99999% of standard Pub/Sub)
  • Zone failure = topic/subscription loss (manual recovery required)
  • For DR: implement cross-zone replication at application level (e.g., Dataflow)
