1. Core Concepts

What is Cloud Functions?

A fully managed, serverless compute service that runs your code in response to events. You write a single function, deploy it, and Google manages the infrastructure—no servers to provision, patch, or scale.

Event-Driven Execution Model

Cloud Functions are triggered by events from Google Cloud services (Pub/Sub, Cloud Storage, Firestore, etc.) or HTTP requests. The function executes in response to an event, completes, and then scales down to zero.

  • No polling required: Events automatically trigger the function
  • Asynchronous by default: Most triggers are async; HTTP triggers are sync
  • Pay-per-invocation: You're charged only for the CPU-seconds consumed during execution
How Cloud Functions Works
  1. You write a function in Node.js, Python, Go, Java, or other runtimes
  2. Deploy it with a trigger (HTTP, Pub/Sub, Cloud Storage, Cloud Firestore, Cloud Tasks, etc.)
  3. When the trigger fires, GCP creates an execution environment and invokes your function
  4. The function runs, processes the event, and returns
  5. The environment scales down; you're billed for CPU-seconds only

Trigger Types

Cloud Functions supports multiple trigger mechanisms. Understanding which to use is critical for the exam.

Trigger Type Activation Invocation Model Best Use Case
HTTP Trigger HTTP/HTTPS request to a public URL Synchronous (caller waits for response) RESTful APIs, webhooks, real-time processing
Pub/Sub Trigger Message published to a Cloud Pub/Sub topic Asynchronous (push subscription) Event streaming, decoupled systems, fan-out processing
Cloud Storage Trigger Object created/deleted/finalized/archived in GCS Asynchronous File processing (images, logs, data files)
Cloud Firestore Trigger Document created/updated/deleted Asynchronous Real-time database synchronization, notifications
Cloud Tasks Trigger Task enqueued in Cloud Tasks queue Asynchronous (push task) Scheduled/delayed execution, task orchestration
Firebase (Realtime DB, Auth, Messaging) Firebase events (database write, auth, remote config) Asynchronous Mobile/Firebase applications
Key Insight: HTTP triggers are synchronous (caller waits for the response), while Pub/Sub, Cloud Storage, Firestore, and Cloud Tasks triggers are asynchronous (function executes in the background, caller doesn't wait).

Generation Comparison: Gen1 vs Gen2

Google released Gen2 (2024) to address Gen1 limitations. The exam may test knowledge of both, but Gen2 is recommended for new deployments.

Aspect Gen1 Gen2
Runtime Custom runtime per Node/Python/Go version Uses Cloud Run container runtime; any image
Concurrency per Instance Only 1 request at a time per instance Up to 1000 concurrent requests per instance
Timeout Max 540 seconds (9 minutes) Max 3600 seconds (1 hour) for HTTP; varies for async
Cold Start Slower cold starts (GCP manages scaling) Built on Cloud Run; faster cold starts
Memory 128 MB to 8 GB 128 MB to 16 GB
Container Image No; source code only Yes; can use custom Dockerfile
Scaling Strategy Instance per request; max_instances limit Requests multiplexed on instances; more efficient
Recommendation Legacy; avoid for new projects Use for all new deployments
EXAM TRAP — Gen1 Concurrency Limitation: Gen1 functions can only handle 1 concurrent request per instance. If you need to handle 100 concurrent requests, GCP will spin up 100 function instances, driving costs up. Gen2 multiplexes up to 1000 requests on the same instance—a massive cost savings.

2. Architecture & Components

Cloud Functions Execution Model (High Level)

Component Role Managed By
Function Code User-written business logic (Python, Node.js, Go, etc.) User
Trigger Event source (Pub/Sub topic, HTTP endpoint, Cloud Storage bucket, etc.) User (configuration) + GCP (routing)
Invocation Service Detects events, routes them to function instances, handles retries GCP (fully managed)
Execution Environment Container/sandbox where the function runs; includes runtime libraries, environment vars GCP (fully managed)
Scaling Controller Creates/destroys instances based on event load and concurrency GCP (fully managed)
Logs & Monitoring Cloud Logging, Cloud Trace, Cloud Profiler integration GCP (automatic)

Event Published/Created
    ↓
Pub/Sub Topic / GCS Bucket detects
    ↓

    ↓
Cloud Functions Invocation Service
    ↓
Check: Is there an available instance?
    ├─ YES: Route to existing instance
    └─ NO: Spin up new instance
    ↓

    ↓

    ↓
If FAILURE and retries enabled: 
    └─ Retry with exponential backoff (default: up to 5 times, 24-hour retry window)
    ↓
Logs written to Cloud Logging
