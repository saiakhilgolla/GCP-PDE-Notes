VPC — GCP PDE Study Notes

Virtual Private Cloud: Core Concepts, Architecture, Configuration, and Exam Patterns

1. Core Concepts

What is a VPC?

A Virtual Private Cloud (VPC) is a logically isolated network within GCP. All resources (Compute Engine, GKE, Cloud SQL, etc.) must reside in a VPC. Each Google Cloud project has a default VPC, but you can create custom VPCs for isolation and control.

Key Fact: VPCs are global resources. Subnets are regional. A single VPC can span multiple regions with subnets in each region.
Subnets

Subnets divide a VPC into regional segments. Each subnet uses a CIDR block (e.g., 10.0.1.0/24). Within a subnet, VMs receive primary and optional secondary IP ranges for alias routing.

  • Primary range: Used for VM primary network interfaces.
  • Secondary range: Used for pod CIDR in GKE or alias IPs for custom routing.
  • Stack type: IPv4, IPv6 Dual Stack, or IPv6-only.
  • Private Google Access (PGA): Allows resources without external IPs to reach Google APIs via private IP.
Firewall Rules

Firewall rules control ingress and egress traffic at the VPC level. Rules are stateful for connections (established responses auto-permit) but rules themselves are stateless (both directions require explicit rules).

  • Direction: INGRESS or EGRESS.
  • Action: ALLOW or DENY.
  • Priority: 0–65535. Lower wins. Default deny is 65534.
  • Match criteria: Source/destination IP, protocol, port.
  • Target: All instances, service accounts, network tags.
  • Deny rules: Explicitly block traffic (useful for policy exceptions).
Exam Trap: Rules DO NOT filter traffic between VMs in the same subnet unless explicitly matched. Use VPC Service Controls (outside VPC scope) for data exfiltration prevention.
Routes

Routes determine how packets are forwarded. Each route has a destination CIDR, next-hop (gateway, peering, VPN), and priority (20 for system default, up to 65535).

  • System routes: Auto-created for subnet ranges and metadata server (169.254.169.254).
  • Custom routes: Point to Cloud VPN, Cloud Interconnect, peering, or a gateway.
  • Default route: 0.0.0.0/0 → default-internet-gateway (priority 1000).
  • Next-hop types: IP address, VPN gateway, Cloud Router, peering connection.
VPC Peering

VPC peering connects two VPCs (in same or different projects, same or different organizations) without routing traffic through the public internet. Low latency, no bandwidth costs, encrypted.

  • Bilateral: Both VPCs must accept the peering for traffic to flow.
  • Transit peering not allowed: If VPC-A peers to VPC-B and VPC-B peers to VPC-C, VPC-A cannot reach VPC-C through VPC-B. Explicit peering required.
  • Transitive routing: Use VPN gateway or Cloud Router to simulate multi-hop routing.
  • Import/export custom routes: Enable flag to share custom routes (e.g., VPN routes) across peered VPCs.
Shared VPC

Shared VPC allows a central admin project to host a VPC. Service projects attach to it and share subnets. Enables centralized network management across teams.

  • Host project: Owns the VPC and subnets.
  • Service project: Attaches to host and launches resources in shared subnets.
  • Subnet-level control: Admin can delegate which subnets service projects can use (per-subnet IAM).
  • Billing: Resources in service projects bill to service project.
  • No subnet sharing cost: Sharing a subnet across service projects is free; each project's resources use it.
Use case: Multi-team enterprise. Finance owns network, engineering and sales teams attach service projects.
Private Google Access (PGA)

PGA enables VMs without external IPs to reach Google APIs and services via a private endpoint over internal routes. Requires enabling PGA on the subnet and routing traffic to the Google VPC.

  • Prerequisite: Subnet must have PGA enabled.
  • Reachable: Compute, Storage, Bigquery, Pub/Sub, Spanner, Cloud SQL (via private IP proxy).
  • Route destination: 199.36.0.0/10 (Google internal range).
  • No external IP needed: VMs securely access Google APIs without NAT or internet gateway.
Private Service Connect

Private Service Connect (PSC) extends Private Google Access. Offers private endpoints for Google services (BigQuery, Cloud Storage, Pub/Sub, Spanner, etc.) and third-party services published via Service Attachment.

  • Service producer: Creates a Service Attachment, exporting a backend service.
  • Service consumer: Creates a Private Service Connect endpoint in their VPC, pointing to the attachment.
  • Replaces VPC peering: No need to peer for API access; endpoint appears as an internal IP.
  • Use case: SaaS provider exposes service via PSC; customer reaches it securely over private link.
VPC Flow Logs

VPC Flow Logs capture network traffic metadata (5-tuple: src/dst IP, port, protocol) for every flow in a VPC or subnet. Records both accepted and denied traffic. Essential for security auditing, troubleshooting, and compliance.

  • Sampling: 1/10,000 by default; tunable (1 = no sampling).
  • Metadata: Source/dest IP, port, protocol, bytes/packets, round-trip time, TCP flags.
  • Logging destination: Cloud Logging, Cloud Storage, BigQuery, Pub/Sub.
  • Cost: Minimal; scales with VPC traffic volume.
  • Enable at: VPC level (all subnets) or per-subnet.
Exam Trap: Flow logs do NOT capture DNS, do NOT see internal firewall denies (only egress denies). Sampling makes high-volume traffic analysis incomplete.

2. Architecture & Components

VPC Topology Reference

Single Region, Multi-Subnet Layout:

Project-A (VPC: prod-vpc)
├── Subnet: us-central1-app (10.0.1.0/24) [PGA enabled]
│   └── VMs, GKE cluster
├── Subnet: us-central1-db (10.0.2.0/24) [PGA enabled]
│   └── Cloud SQL private service
└── Firewall rules:
    ├── allow-ssh-from-iap (0.0.0.0/0 → tcp:22, service account: prod-ssh)
    ├── allow-http (0.0.0.0/0 → tcp:80, network tag: http-server)
    └── deny-db-external (0.0.0.0/0 → 10.0.2.0/24:tcp:3306, priority 100)

Multi-VPC with Peering:

Host Project (VPC: shared-vpc, subnets: us-central1, europe-west1)
├── Service Project-A attached → launches VMs in shared subnets
├── Service Project-B attached → launches GKE in shared subnets
└── VPC Peering to Partner VPC (separate org)
    └── route-exchange enabled (import custom routes)

Separate Org (VPC: partner-vpc, 192.168.0.0/16)
└── Peered to shared-vpc, can reach resources at 10.0.0.0/8
