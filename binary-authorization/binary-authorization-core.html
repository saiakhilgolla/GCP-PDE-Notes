Core Concepts

What is Binary Authorization?

Binary Authorization is a GCP service that enforces deployment of only container images signed by authorized attestors. It provides image integrity verification at deployment time across GKE, Cloud Run, and Anthos.

Core Idea: Before any container image is deployed, Binary Authorization verifies it has been signed by a trusted attestor. Unsigned or untrusted images are rejected automatically.
Key Components
  • Attestor: A trusted entity that signs container images to assert they pass certain criteria (e.g., security scanning, code review)
  • Policy: A declarative rule set defining which images can be deployed based on attestation requirements
  • Attestation: A cryptographic signature proving an image meets certain standards
  • KMS Key: Cloud KMS key used for signing and verifying attestations
  • Breakglass: Emergency allow-list for deployments when normal policy cannot be met
How Binary Authorization Works
  1. Developer pushes container image to Artifact Registry
  2. Image is scanned by Container Analysis (for vulnerabilities)
  3. Attestor (usually automated system) inspects image and signs attestation if it passes criteria
  4. Attestation is stored in Artifact Registry for that image
  5. User attempts deployment to GKE or Cloud Run
  6. Binary Authorization policy engine intercepts deployment request
  7. Engine verifies image has valid attestation(s) from required attestor(s)
  8. If valid, deployment proceeds; if invalid, deployment is rejected

Attestor Types

Type Use Case Key Feature
User-Managed KMS Custom key in Cloud KMS for signing attestations Full control over key lifecycle; requires manual signing
PKIX Standard X.509 certificates for attestation validation Compatible with external PKI systems; chain of trust verification
Policy vs Enforcement

Policy: Declarative rules specifying which attestors are required for specific images or clusters. Written in YAML/JSON.

Enforcement: Binary Authorization service actively intercepts deployment requests and applies the policy in real-time.

Key Difference: Policy is passive configuration; enforcement is active security gate.

Architecture

Component Diagram

Component Role Interaction
Artifact Registry Container image repository Stores images and attestations; triggers Binary Authorization on deploy attempts
Cloud KMS Key management Holds attestor signing keys; used for cryptographic verification
Binary Authorization API Policy enforcement engine Evaluates policy rules against image attestations before deployment
Container Analysis Image scanning (optional) Scans images for vulnerabilities; attestor can require clean scans
GKE Kubernetes cluster Blocked at admission control layer if Binary Authorization denies image
Cloud Run Serverless container platform Blocked at deployment time if Binary Authorization denies image

Deployment Flow

         ↓

         ↓

         ↓

         ↓

    ├─ YES → Deployment Allowed

GKE Integration
  • Admission Controller: Binary Authorization runs as Kubernetes admission webhook on GKE clusters
  • Blocking Point: Intercepts pod creation at kube-apiserver level before kubelet pulls image
  • Per-Cluster Policy: Can assign different policies to different GKE clusters via cluster labels
  • Image Pull: If rejected, pod is never created; image is never pulled from registry
Cloud Run Integration
  • Deployment Time Check: Policy evaluated when service revision is deployed (not at request time)
  • No Cluster Selector: Cloud Run policy is global; cannot have per-service policies
  • Error on Violation: Deployment fails with 403 if attestation requirements not met
  • Buildpacks: Images built via Cloud Build and pushed to Artifact Registry require attestation before deployment
