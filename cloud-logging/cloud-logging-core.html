1. Core Concepts

What is Cloud Logging?

Cloud Logging is GCP's centralized service for collecting, storing, searching, and analyzing logs from all GCP resources, on-premises systems, and third-party applications. Logs flow into a centralized repository called the _Default log bucket where they are retained per default retention policies.
Log Sinks: Route logs to destinations

A sink is a configuration that routes log entries from the source (a project, folder, or organization) to a destination. Sinks filter logs based on search filters and forward matching entries to destinations.

Sink Components:

  • Name: Unique identifier within scope
  • Scope: Project, folder, or organization
  • Filter: Inclusion filter (logs matching this are routed)
  • Destination: Where logs go (Cloud Logging buckets, Cloud Storage, BigQuery, Pub/Sub, Splunk)
  • Service Account: Sink writer identity (must have write permissions at destination)

Key Facts:

  • One sink can route to one destination only
  • Multiple sinks can route to the same destination
  • Sinks are evaluated as inclusion filters (not exclusion)
  • No default sink—must explicitly create to export logs
  • Sink filter is a required field (cannot be empty)
Log Buckets: Storage and retention

A log bucket is a managed storage resource within Cloud Logging that holds log entries. All logs initially flow to the _Default bucket. You can create custom buckets to organize logs by resource, application, or business unit.

Log Bucket Types:

  • _Default bucket: Automatically exists. Retains logs for 30 days (standard) or 7 days (free tier). Immutable retention policy. Cannot be deleted. All logs flow here unless sunk elsewhere.
  • Custom log buckets: Created by user. Configurable retention (1 day to 3650 days). Can have exclusion filters. Can be regional or multi-region. Subject to quotas per project.

Regional vs. Multi-Region Buckets:

  • Multi-region: Default. Replicated across regions for availability. Slightly higher cost.
  • Regional: Stored in one region for data residency compliance. Lower cost. Manually replicate if needed.

Bucket Pricing:

Pay per GB of log data ingested (not stored). _Default bucket ingestion is free for first 50 GB/month per project.

Log Views: Query and analyze logs

A log view is a queryable interface into a log bucket. Each bucket has at least one view (the default immutable view). You can create additional views with different filters over the same bucket data.

View Features:

  • Each view can have its own filter (subset of bucket logs)
  • Query a view using Cloud Logging query UI or Logging API
  • Views are not separate storage—they are filtered views of bucket data
  • Default view in _Default bucket shows all 30-day logs
  • Custom views can filter by severity, resource type, service, etc.
  • Cannot exclude logs from the immutable default view

Use Cases:

  • Create a view for error logs only (severity=ERROR)
  • Create a view for a specific microservice (resource.labels.service_name)
  • Create views for audit logs, security logs, application logs separately
Exclusion Filters: Reduce ingestion costs

An exclusion filter is configured at the bucket level and prevents matching log entries from being ingested into that bucket. Excluded logs never reach the bucket and are not stored.

Key Characteristics:

  • Applied before logs enter the bucket (ingestion time)
  • Excludes logs from that bucket only—does not prevent sinking to other buckets
  • Useful for cost optimization (skip noisy, low-value logs like health checks)
  • Cannot be applied to _Default bucket
  • Cannot be applied to the immutable default view

Common Exclusion Patterns:

  • severity=DEBUG (exclude debug logs)
  • jsonPayload.message=~"health.*check" (exclude health check logs)
  • resource.type="gke_container" AND severity=INFO (exclude GKE info logs)
  • protoPayload.methodName=~"storage.objects.get" (exclude noisy Cloud Storage read ops)
_Default Bucket Deep Dive

The _Default bucket is the system-managed repository for all logs in a GCP project unless explicitly routed elsewhere by sinks.

Properties:

  • Retention: 30 days standard, 7 days free tier (immutable)
  • Location: Multi-region by default
  • Exclusions: Cannot exclude logs from _Default (no exclusion filter support)
  • Deletion: Cannot be deleted or renamed
  • Ingestion: First 50 GB/month free per project

Design Pattern:

To exclude "noisy" logs from storage: Create a custom bucket with exclusion filters and a sink that routes ALL logs there. The _Default bucket still receives all logs (unavoidable), but your custom bucket has a cleaner subset. Alternatively, exclude them at the sink level by not routing them anywhere, but this requires a more complex sink architecture.

Best Practice:

Use _Default for general retention and ad-hoc queries. Route critical logs to dedicated buckets or external destinations (BigQuery, Cloud Storage) for long-term analysis and compliance.
