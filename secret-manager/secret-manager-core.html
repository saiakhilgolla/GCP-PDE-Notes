Core Concepts & How It Works

What is Secret Manager?

Secret Manager is a secure, managed service for storing sensitive data (API keys, passwords, certificates, tokens) with encryption, access control, and audit logging. It handles encryption at rest (CMEK-optional), in transit (TLS), and in memory.

Core Value: Centralized secret storage with fine-grained IAM, automatic rotation, replication, and version management—no hardcoding secrets in code or environment variables.
Secret Versions & Lifecycle

Every secret has versions. When you update a secret, a new version is created automatically. Each version has a state:

  • ENABLED: Current active version, can be accessed
  • DISABLED: Inactive but retained, can be re-enabled
  • DESTROYED: Deleted permanently (immutable once set)

Default: latest enabled version is returned when accessing the secret. You can access specific versions by explicit reference.

gcloud secrets versions access latest --secret=my-secret gcloud secrets versions access v2 --secret=my-secret
Automatic Secret Rotation

Rotation automatically invalidates old versions and creates new ones on a schedule. Useful for credentials that must be rotated periodically (e.g., database passwords every 90 days).

  • Rotation Period: How often rotation triggers (e.g., 30 days)
  • Next Rotation Date: Timestamp of next scheduled rotation
  • Automation: Cloud Scheduler + Cloud Functions trigger rotation logic (custom code)
Key Detail: Secret Manager does NOT auto-rotate values. It schedules and tracks. You provide the Cloud Function that generates and stores the new secret value.
Replication & Geographic Distribution

Secrets can be replicated across regions for high availability and disaster recovery:

  • Automatic Replication: Single region (managed by Google), multi-region (user-managed locations)
  • User-Managed Replication: Explicitly choose replica regions. Charged per region.
  • Read: Any replica can be read. Latency = local replica
  • Write: Always via primary region, then propagates to replicas (~seconds)
TTL & Secret Expiration

Optional time-to-live for automatic secret destruction:

  • Expiration Time: Absolute UTC timestamp when secret is auto-destroyed
  • Effect: Once expired, secret becomes inaccessible. Cannot be recovered.
  • Use Case: Temporary credentials (e.g., session tokens, one-time API keys)
Exam Alert: TTL is set per secret at creation. Once destroyed, no recovery. Plan carefully for critical secrets.
IAM Access Control & Audit Logging

Fine-grained IAM roles control who can read, update, manage secrets:

Role Permissions
secretmanager.secretAccessor Read secret versions (access payload)
secretmanager.secretViewer View secret metadata (not payload)
secretmanager.admin Full control: create, update, delete, rotate
secretmanager.viewer Org-level viewer of all secrets

Audit Logging: All secret access logged to Cloud Audit Logs. READ and WRITE logs captured automatically (DATA_READ logs optional).

Architecture & Components

Architecture Diagram
Component Role Details
Secret Resource Container Named secret (e.g., "db-password"). Holds multiple versions and metadata.
Secret Version Data Unit Individual payload. Each update creates new version. State: ENABLED, DISABLED, DESTROYED.
Primary Region Write Endpoint All writes go here. User chooses at creation. Cannot change later.
Replica Regions Read Endpoints Optional. Replicate data asynchronously from primary (~seconds).
KMS Integration (CMEK) Encryption Key Optional. Use customer-managed KMS key instead of Google-managed encryption.
IAM Bindings Access Control Per-secret or per-project role assignments. Inherited from parent resources.
Cloud Audit Logs Audit Trail Records all access attempts, changes, deletions. Immutable compliance logs.

Data Flow: Reading a Secret

Client (app/service) → IAM Check (verify secretmanager.secretAccessor) → TLS encrypted request to Secret Manager API → Key decryption (KMS or Google-managed) → Secret payload in memory → Return to client over TLS → Audit log recorded

Data Flow: Rotation

Cloud Scheduler (cron trigger) → Cloud Function (custom rotation logic) → Generate new secret value → Call Secret Manager API (AddSecretVersion) → New version created (ENABLED by default) → Old versions can be disabled/destroyed → Audit log recorded
