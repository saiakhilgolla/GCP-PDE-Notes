1. Core Concepts

What is Firestore?

Firestore is a cloud-hosted NoSQL database on Google Cloud Platform. It provides:

  • Real-time synchronization: Clients receive live updates via listeners
  • Automatic scaling: Handles millions of concurrent users
  • Flexible schema: Documents store nested JSON-like data (fields and subcollections)
  • Offline support: SDK caches data locally; syncs when online
  • ACID transactions: Atomicity, Consistency, Isolation, Durability guarantees
  • Two operational modes: Native mode (default) and Datastore mode (legacy)
Documents & Collections

Collection: A group of documents. Collections are created implicitly when you write the first document.

Document: A container of key-value pairs (fields). Each document has a unique ID within its collection.

Field: A key-value pair in a document. Values can be strings, numbers, booleans, timestamps, arrays, maps, or geopoints.

Subcollection: A collection nested inside a document. Path: collection/documentID/subcollection/subDocID

Document ID: Auto-generated by Firestore (e.g., aBc1DeF2GhI3) or you can set custom IDs. IDs must be unique within their parent collection.

Limits:

  • Max document size: 1 MB
  • Max depth of nesting in maps: 20 levels
  • Array elements: Up to 20,000 elements per array field

2. Architecture

Firestore Architecture Overview
Component Role
Client SDK Web, iOS, Android, or backend SDK. Manages local cache, listeners, offline queue.
Firestore Service Serverless managed database. Stores documents, indexes, and metadata.
Real-time Listener WebSocket or gRPC connection pushing live updates to clients on data changes.
Index Service Maintains composite and single-field indexes for query optimization.
Transaction Engine Ensures ACID guarantees. Uses optimistic locking or conflict resolution.
Backup Service Handles on-demand and scheduled backups (separate billing).

Typical Flow:

  1. Client writes document via SDK
  2. SDK routes to Firestore service (REST or gRPC)
  3. Service validates and writes to primary storage
  4. Indexes updated asynchronously
  5. Real-time listeners notified instantly
  6. Changes replicated for durability
Indexes: Single-Field & Composite

Single-field indexes: Created automatically for each field. Enable filtering and sorting on that field alone. No configuration needed.

Composite indexes: Required for queries that filter or sort on multiple fields, or queries with certain operators like array-contains with ordering.

When Firestore requires a composite index:

  • WHERE clause + ORDER BY on different fields
  • Multiple inequality filters on different fields
  • Array-contains with additional filters or ordering
