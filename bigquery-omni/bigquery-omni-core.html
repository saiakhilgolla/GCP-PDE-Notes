BigQuery Omni

1. Core Concepts

Multi-Cloud Data Access

BigQuery Omni extends BigQuery analytics to data stored in AWS S3 and Azure Blob Storage without moving data. Query external data using standard SQL with BigQuery's serverless architecture.

Key Components

  • Omni Connections: Secure connections to external cloud storage
  • External Tables: Metadata pointing to data in S3/Azure
  • Cross-Cloud Queries: Join external data with BigQuery native tables
  • BigLake Tables: Enhanced external tables with row/column-level security

How It Works

  1. Create Connection to AWS/Azure using service accounts/managed identities
  2. Define external table schema pointing to storage location
  3. Query using standard BigQuery SQL
  4. BigQuery processes data in-place without data movement

4. Exam Traps
Common Gotchas

🚨 Authentication Confusion

  • Trap: Assuming BigQuery service account can directly access S3
  • Reality: Must create Connection with cross-account IAM role for AWS
  • Trap: Using storage access keys instead of managed identities for Azure

🚨 Performance Misconceptions

  • Trap: Expecting same performance as native BigQuery tables
  • Reality: External data queries are slower due to network/format overhead
  • Trap: Using CSV for large datasets
  • Reality: Parquet provides much better performance and compression

🚨 Security Assumptions

  • Trap: Thinking external tables inherit BigQuery's security automatically
  • Reality: Need BigLake tables for row/column-level security
  • Trap: Assuming CMEK works with external data
  • Reality: CMEK only applies to BigQuery-managed storage

🚨 Pricing Gotchas

  • Trap: Forgetting about cross-cloud egress charges
  • Reality: Pay for data transfer from AWS/Azure to Google
  • Trap: Assuming caching works like native tables
  • Reality: Limited caching for external data

🚨 Functionality Limits

  • Trap: Trying to use DML (INSERT/UPDATE/DELETE) on external tables
  • Reality: External tables are read-only
  • Trap: Expecting clustering/partitioning options
  • Reality: Must be implemented at storage layer (S3 prefixes, etc.)
5. Quick Cheatsheet

🔗 Connection Setup

-- AWS S3 Connection CREATE CONNECTION aws_connection CONNECTION_TYPE = 'aws' LOCATION = 'us-east1' PROPERTIES = [ ('aws_role_arn', 'arn:aws:iam::123:role/BigQueryRole') ]; -- Azure Connection CREATE CONNECTION azure_connection CONNECTION_TYPE = 'azure' LOCATION = 'us-east1' PROPERTIES = [ ('azure_client_id', 'client-id'), ('azure_tenant_id', 'tenant-id') ];

📊 External Table

CREATE OR REPLACE EXTERNAL TABLE dataset.external_table ( id INT64, name STRING, created_date DATE ) OPTIONS ( format = 'PARQUET', connection_id = 'project.location.connection_name', uris = ['s3://bucket/path/*.parquet'] );

🛡️ BigLake Table

CREATE OR REPLACE BIGLAKE TABLE dataset.biglake_table ( customer_id INT64, email STRING, region STRING ) OPTIONS ( connection_id = 'project.location.connection_name', format = 'PARQUET', uris = ['s3://bucket/data/*.parquet'] );

⚡ Best Practices

  • Use Parquet for best performance
  • Partition data at storage layer
  • Use BigLake for security policies
  • Minimize cross-cloud data transfer
  • Cache frequently accessed data in BigQuery
