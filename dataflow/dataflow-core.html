Dataflow Study Notes

Core Concepts

Apache Beam Programming Model

  • Pipeline: Represents the entire data processing workflow
  • PCollection: Immutable, distributed data set that flows through pipeline
  • Transform: Processing step that takes PCollections as input and produces PCollections as output
  • I/O Connectors: Read from and write to external systems (BigQuery, Pub/Sub, Cloud Storage)

Streaming vs Batch Processing

AspectBatchStreaming

Windowing Concepts

  • Fixed Windows: Non-overlapping time intervals (e.g., every 5 minutes)
  • Sliding Windows: Overlapping time intervals (e.g., 10-minute window every 5 minutes)
  • Session Windows: Based on data activity with timeout gaps
  • Global Window: Single window containing all data (default for batch)

Watermarks & Late Data

  • Watermark: Estimate of when all data up to a certain event time has been seen
  • Event Time: When the data event occurred
  • Processing Time: When Dataflow processes the event
  • Late Data: Data arriving after the watermark has passed
  • Allowed Lateness: How long to wait for late data before finalizing results

Triggers

  • Default Trigger: Fire when watermark passes end of window
  • Processing Time Trigger: Fire at regular processing time intervals
  • Data-driven Trigger: Fire based on data characteristics (element count, size)
  • Composite Triggers: Combine multiple trigger conditions (early, on-time, late)

Execution Model

  • Runner: Dataflow Runner translates Beam pipeline to execution graph
  • Stage: Collection of transforms that can run on same worker
  • Fusion: Combines multiple transforms into single stage for efficiency
  • Work Item: Unit of parallel processing within a stage

Streaming Engine Architecture

  • Separates compute (workers) from storage (managed state)
  • State stored in persistent backend, not local worker disk
  • Enables faster autoscaling and worker replacement
  • Required for certain features like Exactly Once Pub/Sub processing

Exam Traps

Common Gotchas

  • Streaming Engine Required: For exactly-once Pub/Sub processing, not optional
  • Windowing Confusion: Batch uses Global window by default, streaming requires explicit windowing for aggregations
  • Late Data Handling: Default trigger only fires at watermark - need composite triggers for early results
  • Template Types: Classic templates can't install custom dependencies - need Flex templates
  • Regional Limits: Workers are regional resource - can't span multiple regions in same job

Performance Traps

  • Hot Keys: Avoid keys that concentrate too much data - causes worker bottlenecks
  • Side Inputs: Large side inputs loaded on every worker - use broadcast pattern carefully
  • Fusion Breaks: External I/O, GroupByKey, and Flatten operations break fusion
  • Window Size: Very small windows create overhead, very large windows increase memory usage
  • Machine Types: n1-highmem for memory-intensive, c2 for CPU-intensive workloads

Cost Optimization Traps

  • Preemptible Workers: Only for batch jobs, not streaming (workers can be interrupted)
  • Streaming Engine Cost: Additional cost but enables better scaling and features
  • Over-provisioning: Setting max workers too high leads to unnecessary scaling costs
  • Regional Resources: Choose region close to data sources to minimize egress charges

Security & Compliance Traps

  • Service Account: Workers use Compute Engine default SA unless specified - follow least privilege
  • VPC Networks: Default network allows all internet access - use custom networks for isolation
  • Private Google Access: Required when workers have no external IP addresses
  • CMEK Support: Available for persistent disks but not for Shuffle service data
Quick Cheatsheet

When to Use Dataflow

  • ✅ ETL with complex transformations
  • ✅ Real-time stream processing
  • ✅ Batch data processing at scale
  • ✅ Need autoscaling and managed infrastructure
  • ❌ Simple SQL transformations (use BigQuery)
  • ❌ Small datasets (<1GB)
