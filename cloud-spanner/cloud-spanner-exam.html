Core Concepts

What is Cloud Spanner?

Globally distributed, horizontally scalable, ACID-compliant relational database. Combines SQL semantics with NoSQL scale.

TrueTime

Google's globally synchronized clock system that provides external consistency. Ensures transactions appear to execute in chronological order globally.

Key Features

  • External consistency: Linearizable reads/writes across regions
  • Auto-sharding: Automatic horizontal partitioning
  • 99.999% availability: Multi-region instances
  • SQL interface: Standard SQL with extensions

🚨 Exam Traps

Hot Spot Traps

  • Sequential primary keys: Auto-incrementing IDs create hot spots. Use GENERATE_UUID() or reverse timestamp
  • Timestamp-based keys: All writes go to latest partition. Prefix with random component
  • Monotonic keys trap: Don't use sequential customer_id as first primary key column

Schema Design Traps

  • Interleaving misconception: Parent-child rows are co-located but you can still query child table independently
  • Foreign keys don't exist: Spanner has no foreign key constraints. Use interleaving for parent-child relationships
  • Array size limit: Arrays can be very large but impact performance. Consider separate table for large arrays

Transaction Traps

  • Read-write vs read-only: Read-only transactions are cheaper and can use stale reads. Specify explicitly
  • Stale reads confusion: strong=false gives bounded staleness (~10s), not exact staleness
  • Cross-region latency: Multi-region writes have higher latency due to Paxos consensus
  • Transaction size: Large transactions (>20K mutations) should be batched

Multi-Region Traps

  • Default leader placement: Spanner chooses leader region automatically. Use default_leader option for predictable latency
  • Read replicas vs multi-region: Read replicas are for read scaling in single region. Multi-region is for global distribution
  • Configuration naming: Multi-region configs have specific names like nam6, not custom names

Performance Traps

  • Node scaling: Nodes don't auto-scale by default. Must configure auto-scaling or scale manually
  • CPU utilization target: Keep below 65% for multi-region, 45% for regional instances
  • Query patterns: Avoid large table scans. Use proper indexing and LIMIT clauses
⚡ Quick Cheatsheet

Primary Key Design

  • ❌ AUTO_INCREMENT, sequential IDs, timestamp-first
  • ✅ GENERATE_UUID(), hash-based, composite with random prefix
  • ✅ Use customer_id as SECOND column in composite key

Interleaving

  • Syntax: INTERLEAVE IN PARENT ParentTable ON DELETE CASCADE
  • Benefit: Parent-child rows stored together, single-transaction updates
  • Limit: Max 7 levels deep
  • Use when: Always query parent+child together

Transaction Types

  • Read-write: Default, supports mutations, uses latest timestamp
  • Read-only: Cheaper, can use stale reads, specify ReadOnlyTransaction()
  • Stale reads: TimestampBound.ofExactStaleness(15, TimeUnit.SECONDS)

Scaling

  • Regional: 1-2,000 nodes, 99.99% SLA
  • Multi-region: 1-10,000 nodes, 99.999% SLA
  • Processing Units: 100-10,000 PU (1000 PU = 1 node)
  • Auto-scaling: Available for PU-based instances

Common DDL

  • CREATE TABLE - Standard SQL syntax
  • CREATE INDEX - Secondary indexes
  • ALTER TABLE ADD COLUMN - Schema evolution
  • DROP TABLE - Cannot drop parent table with children
