Cloud Spanner Study Notes

1. Core Concepts

What is Cloud Spanner?

Globally distributed, horizontally scalable, strongly consistent relational database. Combines RDBMS structure with NoSQL scale.

TrueTime API

TrueTime provides globally synchronized timestamps with bounded uncertainty. Each server gets a time interval [earliest, latest] rather than exact time. Spanner waits for uncertainty to pass before committing transactions, ensuring global consistency.

Key Features

  • Horizontal Scaling: Add nodes to increase throughput and storage
  • Strong Consistency: ACID transactions across all regions
  • Global Distribution: Multi-region with automatic failover
  • SQL Interface: Standard SQL with extensions for distributed queries
  • Schema Changes: Online schema updates without downtime

Interleaving

Table Interleaving: Child table rows are physically stored with parent rows. Parent-child rows on same split = single-node transactions. Example: Orders interleaved in Customers table.

4. Exam Traps

Hot Spots

Trap: Sequential primary keys (auto-increment IDs, timestamps) create hot spots. All writes go to one split.

Solution: Use UUID, hash of timestamp, or reverse timestamp as first part of primary key.

Primary Key Design

Trap: Bad primary key = poor distribution. First column determines data distribution.

Best Practice: High cardinality first column, avoid monotonically increasing values.

Interleaving Gotchas

  • Wrong: Interleave unrelated tables to "improve performance"
  • Right: Only interleave genuine parent-child relationships
  • Trap: Too much interleaving can create large splits

Transaction Boundaries

Trap: Cross-region transactions are expensive. Design schema to minimize cross-region reads/writes.

Schema Changes

Remember: Most schema changes are online and non-blocking, but adding NOT NULL columns or changing primary keys requires table recreation.

5. Quick Cheatsheet

Schema Design

  • Primary key = data distribution
  • First column = highest cardinality
  • Avoid sequential keys
  • Interleave parent-child only
  • Secondary indexes for non-key queries

Performance

  • 1 node = 1,000 processing units
  • 2,000-10,000 QPS per node
  • Regional: 99.99% SLA
  • Multi-regional: 99.999% SLA
  • Stale reads = lower latency

SQL Extensions

  • INTERLEAVE IN PARENT
  • @{FORCE_INDEX=index_name}
  • TABLESAMPLE SYSTEM (10 PERCENT)
  • Array and struct support
  • Commit timestamps

Common Mistakes

  • Timestamp as first primary key column
  • Over-interleaving tables
  • Cross-region transaction patterns
  • Not using stale reads for analytics
  • Ignoring hot spot warnings
