Vertex AI Feature Store - GCP PDE Study Notes

1. Core Concepts

Feature Store Fundamentals

Vertex AI Feature Store is a centralized repository for ML features that provides consistent feature serving for training and prediction with point-in-time correctness.

Component Description Use Case
Feature Store Top-level container for feature groups Organization boundary (one per project/region)
Feature Group Collection of related features with shared schema Group features from same data source/entity
Feature Individual ML feature with metadata Single predictive attribute
Feature View Subset of features from one/multiple groups Model-specific feature selection

Key Workflows

Ingestion: Batch via BigQuery → Feature Store, Stream via Pub/Sub → Dataflow → Feature Store
Serving: Online (real-time predictions) vs Offline (batch training)

Point-in-Time Correctness

Ensures training features match serving features by maintaining temporal consistency. Prevents data leakage by only using feature values available at prediction time.

4. Exam Traps

⚠️ Common Pitfalls

BigQuery ML vs Feature Store Latency

Trap: Using BigQuery ML.PREDICT for real-time serving with <100ms requirements

Reality: BigQuery ML.PREDICT has seconds-level latency. For <100ms, export model to Vertex AI and use Feature Store online serving

Online vs Offline Serving Confusion

Trap: Using offline serving for real-time predictions

Reality: Online = Bigtable (<100ms), Offline = BigQuery (batch training only)

Feature Group vs Feature View

Trap: Creating separate Feature Groups for each model

Reality: Feature Groups organize by data source/entity. Use Feature Views for model-specific subsets

Point-in-Time Correctness

Trap: Ignoring temporal alignment between training and serving

Reality: Always specify timestamps to prevent data leakage

Regional Limitations

Trap: Assuming global availability

Reality: Feature Store is region-specific. Data cannot cross regions

Bigtable Backend Lock-in

Trap: Expecting to access underlying Bigtable directly

Reality: Bigtable is managed by Feature Store. Access only via Feature Store APIs

Decision Traps

Wrong Choice Right Choice When
BigQuery for online serving Feature Store online serving Latency < 1 second required
Vertex AI Pipelines for feature ingestion Direct BigQuery import Simple batch ETL from BigQuery
Cloud SQL for feature storage Feature Store ML workloads need point-in-time correctness
Custom Redis cache Feature Store online serving Need managed low-latency feature serving
5. Quick Cheatsheet

🎯 Key Decision Points

When to Use Feature Store

  • Real-time ML predictions with <100ms latency
  • Need point-in-time correctness for training
  • Multiple models sharing features
  • Feature reuse across teams

Architecture Patterns

  • BigQuery → Feature Store → Online Serving (batch + real-time)
  • Pub/Sub → Dataflow → Feature Store (streaming ingestion)
  • Feature Store → Vertex AI Training (offline serving)
  • Feature Store → Vertex AI Endpoints (online serving)
