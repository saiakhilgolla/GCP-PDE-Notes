1. Core Concepts

What is Cloud KMS?

Cloud Key Management Service (KMS) is a managed encryption service for creating, rotating, and managing cryptographic keys. Used to encrypt sensitive data at rest using customer-managed encryption keys (CMEK).

Key Hierarchies: KEK vs DEK

KEK (Key Encryption Key) vs DEK (Data Encryption Key)

KEK (Key Encryption Key):

  • Master key stored in Cloud KMS
  • Used to encrypt other keys
  • Never leaves KMS (except for manual export with export policies)
  • Managed by you or Google
  • Rotation handled by KMS

DEK (Data Encryption Key):

  • Data encryption key that actually encrypts your application data
  • Encrypted by KEK
  • Can exist in application memory or storage
  • Unique per data object (often)
  • Generated on-demand or in bulk

Pattern: Data encrypted with DEK → DEK encrypted with KEK → KEK stored in KMS

Envelope Encryption Pattern

How envelope encryption works

Step 1: Generate DEK

  • Request plaintext DEK from Cloud KMS
  • Receive both plaintext and encrypted versions

Step 2: Encrypt Data

  • Use plaintext DEK to encrypt your data
  • Store encrypted data + encrypted DEK together
  • Discard plaintext DEK from memory

Step 3: Decrypt Data

  • Fetch encrypted DEK from storage
  • Call KMS to decrypt the DEK using KEK
  • Use plaintext DEK to decrypt data

Benefit: KMS doesn't see or encrypt your actual data — only the small DEK. Reduces KMS throughput bottleneck.

CMEK (Customer-Managed Encryption Keys)

CMEK means you create and control the encryption key in Cloud KMS. Google services use your key to encrypt their managed services (BigQuery, Cloud Storage, Pub/Sub, etc.). Alternative: Google-managed keys (GMEK) where Google controls key generation and rotation.

Key Rings

Understanding Key Rings
  • Key Ring: A container that holds cryptographic keys in a specific location
  • Regional scope: Key rings exist in a GCP region (us-central1, europe-west1, etc.)
  • Organization: Group related keys in a key ring (e.g., all keys for a project)
  • No resource hierarchy: Key rings don't follow folder/project nesting — they're independent resources at regional level
  • Access control: IAM bindings apply to individual key rings and keys within them
  • Cannot be deleted: Once created, key rings cannot be deleted (can only be managed)
  • Operations: You cannot encrypt/decrypt with a key ring directly — only with a key version inside it

Key Versions and Rotation

Key versions explained
  • Key Version: A specific instance of a cryptographic key with a unique version number (1, 2, 3...)
  • Primary version: The key version used by default for encrypt operations
  • Rotation: Process of creating a new key version and promoting it to primary
  • Asymmetric rotation: Old versions remain active for decryption (backward compatibility)
  • Automatic rotation: KMS can automatically create new versions on a schedule (90 days default)
  • Manual rotation: You can create versions on demand
  • Version states: PENDING_GENERATION, ENABLED (can encrypt/decrypt), DISABLED, DESTROYED
  • Destroyed versions: Data encrypted with destroyed versions cannot be recovered (permanent)
  • Max 1000 versions: A single key can have up to 1000 versions

HSM vs Software Keys

Hardware Security Module keys vs software-based keys
Aspect HSM Keys Software Keys
Storage Dedicated hardware security module Google-managed secure servers
Compliance FIPS 140-2 Level 3, cryptographic validation, key never leaves HSM FIPS 140-2 Level 2, sufficient for many use cases
Cost Higher per-key-version Lower cost
Latency Slightly higher (physical crypto) Lower latency
Use Cases Regulated industries (finance, healthcare), strict compliance, key material cannot leave jurisdiction General encryption, most GCP services, cost-sensitive
Key Export Cannot export (enforced by HSM) Can export with proper IAM permissions
Available Operations Encrypt, Decrypt, Sign, Verify, Asymmetric Decrypt All cryptographic operations
