1. Core Concepts

What is VPC Service Controls?

VPC Service Controls is a security feature that creates a security perimeter around Google Cloud services. It prevents data exfiltration by enforcing access policies at the service boundary, not just through IAM credentials.

Key Insight: Blocks requests from outside the perimeter even if the requester has valid IAM credentials. This protects against compromised credentials and insider threats.

Core Problem Solved: Traditional IAM only checks who you are. VPC Service Controls also checks where you are (network location) and the context of the request.

Service Perimeters

A service perimeter is a logical boundary that defines which resources and services are protected by VPC Service Controls.

  • Regular Perimeter: Protects resources within the perimeter. Requests from outside are blocked.
  • Egress Rules: Define allowed destinations for traffic leaving the perimeter.
  • Ingress Rules: Define allowed sources for traffic entering the perimeter.
Default Behavior: All requests from outside the perimeter are denied unless explicitly allowed by an ingress rule.
Access Policies and Access Levels

Access Policy: Top-level object that contains access levels and service perimeters. One per organization.

Access Levels: Named conditions that define criteria for allowing access. Examples:

  • IP ranges (allow traffic from corporate network)
  • Device criteria (only managed devices)
  • User identity (specific user or group)
  • Time-based criteria (business hours only)
Relationship: Perimeters use Access Levels as conditions in ingress/egress rules.
Bridge Perimeters

A special type of perimeter that allows communication between two separate regular perimeters without exposing resources to the public internet.

  • Connects two or more perimeters
  • Only supports ingress/egress to other perimeters
  • Does not protect resources directly
  • Useful for cross-team or multi-environment setups
Use Case: Team A has perimeter with datasets. Team B has perimeter with BigQuery ML. Use bridge to allow Team B's ML to read Team A's data without making data public.
Dry Run Mode

A testing mode for perimeters where violations are logged but not enforced.

  • Requests are allowed to proceed normally
  • All violations are recorded in Cloud Audit Logs
  • No impact on running applications
  • Essential for testing before enforcement
Best Practice: Always deploy perimeters in dry run mode first. Monitor logs for 1-2 weeks. Then switch to enforced mode.
Supported Services

VPC Service Controls works with Google Cloud managed services (not compute instances). Supported services include:

  • Cloud Storage
  • BigQuery
  • Cloud Spanner
  • Cloud Datastore
  • Pub/Sub
  • Cloud SQL
  • Cloud Firestore
  • Cloud KMS
  • And 20+ others
TRAP: Compute Engine and GKE are NOT protected by service perimeters. Use VPC firewall rules for those instead.

2. Architecture & Components

VPC Service Controls Architecture

Component Role Key Details
Access Policy Container for all controls Organization-level. Contains 1+ perimeters and access levels.
Access Level Define conditions for access Reusable. Evaluated at request time. Can be combined with OR logic.
Regular Perimeter Protect resources Contains 1+ services. Blocks external requests by default.
Bridge Perimeter Connect perimeters No protected resources. Routes traffic between regular perimeters.
Ingress Rule Control inbound traffic Allows traffic from sources matching access levels.
Egress Rule Control outbound traffic Allows traffic to destinations matching access levels.
Request Flow Through Perimeter
  1. Request arrives at Google Cloud service (e.g., Cloud Storage)
  2. Service checks if resource is in a perimeter
  3. If yes, evaluate ingress rules:
    • Is request source in allowed access level?
    • Does source IP match whitelist?
    • Is requestor using managed device?
  4. If ingress check passes, check egress:
    • Can this service access external resources (if applicable)?
  5. If all checks pass, request proceeds. Otherwise, return 403 Forbidden.
Timing: Checks happen at the service boundary, before IAM evaluation. A request can have valid IAM but fail the perimeter check.
