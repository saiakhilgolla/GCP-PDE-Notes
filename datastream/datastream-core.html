1. Core Concepts

What is Datastream?

Datastream is a fully managed, serverless CDC (Change Data Capture) service that replicates data from source systems into Google Cloud destinations in real-time. It enables continuous data synchronization without building custom replication pipelines.

  • Real-time CDC: Captures INSERT, UPDATE, DELETE events as they occur
  • Managed service: No infrastructure to provision or manage
  • Low latency: Sub-second to seconds depending on source and configuration
  • Data consistency: Transactional consistency guarantees maintained
Change Data Capture (CDC) Explained

CDC captures and tracks changes (inserts, updates, deletes) happening in a database and streams those changes to downstream systems.

How Datastream Implements CDC:

  • Query logs: Reads database transaction logs (binlog for MySQL, WAL for PostgreSQL, redo logs for Oracle)
  • No triggers: Doesn't use application-level triggers; source database handles it natively
  • Ordering preserved: Changes maintain source database transaction order
  • Deduplication: Handles network retries without duplicating events
Backfill vs. CDC Streaming

Datastream operates in two phases:

Phase Purpose Scope Speed
Backfill Initial snapshot of existing data Historical data at stream creation time High throughput bulk copy
CDC Streaming Ongoing changes after backfill completes INSERT, UPDATE, DELETE operations only Low latency real-time
Key Point: Backfill and CDC run in parallel by default. CDC can start capturing changes while backfill is still running, ensuring no data loss.
Stream Types
Stream Type Includes Use Case
Full Refresh Backfill + CDC New stream setup; sync all historical + ongoing changes
CDC Only Changes only (no backfill) Stream already has data; capture changes after point-in-time

Supported Destinations
  • BigQuery: Data Lake warehouse; ideal for analytics
  • Cloud Storage (GCS): JSON newline-delimited format; use for Dataflow/Spark pipelines
  • Firestore in Datastore mode: NoSQL; good for operational data sync
  • Cloud SQL (MySQL/PostgreSQL): Replicate to other databases for HA/DR
Latency Characteristics
Typical latencies:
  • Sub-second (< 1 sec): Low transaction volume on small tables
  • 1-5 seconds: Normal OLTP workloads with moderate change rate
  • 5-30+ seconds: High-volume writes, large backfill, network delays

Factors affecting latency:

  • Source database commit rate (writes/sec)
  • Network connectivity quality (GCP to source)
  • Destination throughput limits (BigQuery slots, Cloud SQL)
  • Transaction log lag on source
  • Backfill progress (if still running in parallel)

2. Architecture

Datastream Architecture Components
Component Function Location
Source Connector Reads logs from source DB; handles connection, authentication, log parsing On-premise or Cloud SQL
Replication Server Deserializes log events; applies filtering; batches changes; maintains ordering Google-managed in GCP region
Destination Writer Writes events to BigQuery/GCS/Firestore; handles retries and backpressure Google-managed in GCP region
Metadata Store Tracks stream state, checkpoints, schema, errors Google-managed
Data Flow: Backfill Phase
  1. Datastream connects to source database with read-only credentials
  2. Queries source tables directly for full snapshot (SELECT * FROM table)
  3. Batches rows and sends to destination (BigQuery/GCS) in parallel threads
  4. Records checkpoint progress in metadata store
  5. If interrupted, resumes from last checkpoint (idempotent)
Backfill uses table snapshots, not transaction logs. It reads consistent point-in-time state of each table.
